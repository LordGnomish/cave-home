**Summary.** cave-home-core is a Rust port of Home Assistant's core.py synchronous data spine (Context, Event, EventBus, State, StateMachine, ServiceRegistry). It implements 21 tests covering the full synchronous contract. Missing behaviors are primarily deferred async-runtime features (HomeAssistant loop/executor, async_call execution, HassJob machinery) requiring Phase-1b runtime layer, plus two Phase-2 ergonomics refinements (listener cancellation handles, State helper accessors).

## Missing behaviors

### `ha-runtime-home_assistant` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

HomeAssistant runtime object: event loop ownership, executor management, start/stop lifecycle, bus/state/service facades. Per Charter §5.1, the async runtime loop hosting belongs to the cave-runtime integrations layer, not here.

- **Spec:** homeassistant/core.py::HomeAssistant class (async event loop, executor, bus/states/services properties, async_create, async_block_till_done, async_run)
- **Notes:** Intentionally deferred to runtime layer; this crate ships the loop-free synchronous spine (Context, Event, EventBus, State, StateMachine, ServiceRegistry) that it will be hosted in. Requires async runtime host.

### `ha-service-async_call` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

ServiceRegistry.async_call: execution path scheduling registered service handler coroutine on event loop, target/entity_id selector expansion, blocking/return_response honour, job scheduling. ServiceCall + Service descriptor already implemented here.

- **Spec:** homeassistant/core.py::ServiceRegistry.async_call(domain, service, service_call, blocking, return_response)
- **Notes:** Execution layer is bound to the async runtime; deferred with HomeAssistant. Registry + ServiceCall data model (register/remove/has_service/services) is fully implemented and tested; execution endpoint is unimplemented.

### `ha-hassjob-callbacks` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

HassJob + async_add_job / async_add_executor_job: coroutine/callback/executor-job dispatch primitives. Wraps user functions (coroutines, blocking callables) for scheduling on the event loop.

- **Spec:** homeassistant/core.py::HassJob class, HomeAssistant.async_add_job, async_add_executor_job, run_callback_threadsafe
- **Notes:** Inseparable from HomeAssistant async loop; deferred to Phase 1b. User-facing service handlers and automations will be wrapped via this machinery.

### `ha-config-runtime` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Config object: runtime configuration (config_dir, location, units, allowlist, etc.) and config_entries layer. cave-home's own config layer (Charter §5) supersedes this.

- **Spec:** homeassistant/core.py::Config class, config_entries module
- **Notes:** Runtime configuration belongs to cave-home's own config subsystem (Charter §5 design). HA Config surface lands in Phase 1b integration layer. No voluptuous/YAML schema porting.

### `ha-lifecycle-events` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Lifecycle events: EVENT_HOMEASSISTANT_START, _STOP, _STARTED, _FINAL_WRITE + CoreState enum. Fired by runtime as it boots/shuts down. Require HomeAssistant runtime object.

- **Spec:** homeassistant/const.py::EVENT_HOMEASSISTANT_START, EVENT_HOMEASSISTANT_STOP, EVENT_HOMEASSISTANT_STARTED, EVENT_HOMEASSISTANT_FINAL_WRITE; homeassistant/core.py::CoreState
- **Notes:** Meaningless without runtime loop; deferred to Phase 1b. Fired from HomeAssistant.async_block_till_done and loop lifecycle hooks.

### `ha-event-call_service-flow` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

EVENT_CALL_SERVICE event: fired on bus when a service is dispatched. AsyncServiceRegistry.async_call_from_config: config-driven service-call path. Both are part of execution layer.

- **Spec:** homeassistant/core.py::ServiceRegistry.async_call_from_config, HomeAssistant fires EVENT_CALL_SERVICE
- **Notes:** Part of async execution layer deferred with async_call to Phase 1b. Registry + ServiceCall data model present; call dispatch is unimplemented.

### `ha-listener-cancel-handle` — phase-2 — ✅ TDD-ready

EventBus listener cancellation: explicit cancel-by-id handle (CALLBACK_TYPE unsubscribe function return). Currently cancels by dropping receiver; Phase-2 adds ergonomic cancel() method on ListenerId.

- **Spec:** homeassistant/core.py::EventBus.async_listen returns CALLBACK_TYPE (async_remove callable)
- **Notes:** Ergonomics refinement; listener table and drop-based cleanup already work. Phase-2 adds ListenerId::cancel(bus) or similar to unsubscribe without owning the receiver. Purely additive, no protocol change.

### `ha-state-helper-accessors` — phase-2 — ✅ TDD-ready

State helper accessors: as_dict() (serialize to dict), domain / object_id (extract from entity_id), name (user-facing friendly name from attributes). Pure derivations over existing fields.

- **Spec:** homeassistant/core.py::State.as_dict, State.domain, State.object_id, State.name property
- **Notes:** Convenience projections; all underlying data (entity_id, attributes, timestamps) present. Phase-2 adds these as impl methods on State. No new data or semantics needed.
