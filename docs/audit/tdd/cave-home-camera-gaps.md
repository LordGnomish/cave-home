# TDD gap analysis — `cave-home-camera`

- **Upstream:** blakeblackshear/frigate
- **License class:** gpl-agpl-cleanroom
- **Port method:** spec-based (detection-policy core); clean-room reimplementation using documented behavior and standard algorithms (ray-casting point-in-polygon, IoU, greedy IoU association), NOT ported from AGPL source
- **Existing cave tests (agent count):** 63
- **Missing behaviors identified:** 11 (11 strict-TDD-ready)

> Generated by the spec-based gap-analysis pass (read-only). For
> `gpl-agpl-cleanroom` upstreams no upstream source/tests were consulted —
> behaviors are derived from public protocol/spec knowledge only.

**Summary.** cave-home-camera is a well-tested Phase-1 MVP implementing the Frigate-class detection decision core (zones, tracking, clip policy, retention) as pure logic with no ML/network/video dependencies. The 63 existing tests cover the main happy paths thoroughly. Missing behaviors center on defensive edge cases, multi-zone interactions, record mode integration, and label-specific retention semantics that are spec-defined but lack explicit test coverage.

## Missing behaviors

### `multi-zone-same-frame` — phase-1b — ✅ TDD-ready

A single detection accepted by multiple zones in the same frame is counted separately in each zone's activity, producing independent clip start/continue actions per zone

- **Spec:** Frigate documented behavior: per-zone detection filtering + independent zone clip policies (parity.manifest.toml §detection filter pipeline + per-zone clip policy); zone.rs::filter returns detections per zone independently
- **Notes:** Already implemented (zone.rs::filter is per-zone, policy.rs::ClipPolicy tracks per-zone), but no explicit test verifies that one detection in multiple zones triggers independent clip actions. Candidate test: assert that passing the same Detection through two Zone::accepts calls, then two separate ClipPolicy::observe calls, produces independent Start actions.

### `continuous-record-mode` — phase-1b — ✅ TDD-ready

In RecordMode::Continuous, the decision logic signals continuous recording intent (all frames recorded, not just motion-triggered clips)

- **Spec:** Frigate documented record retain modes: 'all' / 'motion' / 'active_objects' (parity.manifest.toml spec_sources); config.rs::RecordMode enum
- **Notes:** RecordMode::Continuous is defined but the policy layer (ClipPolicy) does not interact with it — that is deferred to Phase 1b (the muxer). A test would verify that a CameraConfig::record_mode() getter correctly returns Continuous and that records_anything(Continuous) == true. This is more of a config-validation test than behavioral logic.

### `tracker-max-age-saturating` — phase-1b — ✅ TDD-ready

Tracker respects max_age timeout with saturating arithmetic such that a non-monotonic 'now' (backwards clock) never causes false track retirement

- **Spec:** src/track.rs::Tracker::update() uses age() with saturating_sub; TrackedObject::age(now) saturates at 0
- **Notes:** Already implemented (track.rs uses saturating_sub, tests exist for gap survival), but no explicit test covers the saturating behavior itself. Test: create a track at tick 100, skip to tick 50 (backwards), verify track still exists (age saturates to 0, stays < max_age).

### `clip-policy-pre-roll-zero` — phase-1b — ✅ TDD-ready

ClipPolicy with pre_roll=0 starts a clip exactly at the first activity frame with no lead-in padding

- **Spec:** policy.rs::ClipPolicy::observe() returns ClipAction::Start(now.saturating_sub(self.pre_roll)); pre_roll=0 → Start(now)
- **Notes:** Implemented and partially tested (clip_start_pre_roll_saturates_at_zero tests saturation, not zero-roll). Explicit test: assert ClipPolicy::new(0, 10).observe(true, 100) == ClipAction::Start(100).

### `detection-label-all-values` — phase-1b — ✅ TDD-ready

All 7 ObjectLabel variants (Person, Car, DeliveryVan, Bicycle, Dog, Cat, Package) are recognized and produce correct zone filtering, anchor calculation, and localized output in all 3 languages

- **Spec:** label.rs::ObjectLabel::ALL; each label has noun() in EN/DE/TR; seen_at() composes label + place; zone.rs::Zone::wants_label checks membership
- **Notes:** Coverage is high (every_label_has_a_noun_in_every_language exists), but no test explicitly verifies that each of the 7 labels works end-to-end through zone filtering and clip policy together. Candidate: parametrized test that creates a zone watching each label, passes detection of each label, verifies accepts() result.

### `debounce-label-isolation` — phase-1b — ✅ TDD-ready

Debounce correctly isolates quiet windows per label: a Person alert at tick 0 does not suppress a Cat alert at tick 5 even if the Cat quiet window is 30s

- **Spec:** policy.rs::Debounce::allow() maintains per-label quiet history (last_fired Vec); each label has independent clock
- **Notes:** Implemented and tested (each_label_has_its_own_quiet_window exists), confirming no gap here. Listed for completeness.

### `zone-friendly-name-unlocalized` — phase-1b — ✅ TDD-ready

A zone with an unrecognized name (e.g., 'garage_entry') falls back to the readable form (underscores → spaces) in all 3 languages

- **Spec:** zone.rs::Zone::friendly_name() matches known zones, falls back to name.replace('_', ' '); ADR-007 / Charter §6.3 grandma-friendly output
- **Notes:** Implemented and tested (friendly_name_known_and_fallback exists), confirming no gap. Listed for completeness.

### `tracker-id-monotonic-growth` — phase-1b — ✅ TDD-ready

Tracker assigns monotonically increasing TrackId values such that older tracks never receive higher IDs than newer tracks within the same session

- **Spec:** track.rs::Tracker maintains next_id u64 field and increments with saturating_add; each new track gets a unique increasing ID
- **Notes:** Implemented (track.rs increments next_id), but no explicit test verifies monotonicity across multiple track creations. Test: create 3 detections, update tracker, verify IDs are distinct and ordered.

### `camera-config-enabled-labels-empty-means-watch-all` — phase-1b — ✅ TDD-ready

A CameraConfig with no enabled labels (empty vec) means the camera is not actively watching for anything until labels are configured; querying watches() returns false for all labels

- **Spec:** config.rs::CameraConfig::watches() returns self.enabled_labels.contains(&label); empty enabled_labels → always false
- **Notes:** Implemented and tested (fresh_camera_has_safe_defaults verifies empty enables watch nothing). The zone-level behavior (empty required_labels means any) is different and is tested. Listed for clarity.

### `geometry-degenerate-polygon-test-coverage` — phase-1b — ✅ TDD-ready

Polygon with exactly 3 points (minimum) successfully contains points and rejects invalid membership tests; Polygon with 1000+ points (stress) still computes correctly

- **Spec:** geometry.rs::Polygon::new() requires >= 3 points; contains() uses O(n) ray-casting
- **Notes:** Minimum case (3-point triangle) is tested implicitly in zone tests. Stress case (large polygon) is not tested. Candidate: explicit test with a 100+ vertex polygon.

### `retention-per-label-classification` — phase-2 — ✅ TDD-ready

classify_retention() applies the same retention window to all labels, but CameraConfig can be extended per-label in Phase 1b; test current single-window behavior

- **Spec:** policy.rs::classify_retention(recorded_at, now, retention_days) uses a single window for all events; config.rs::CameraConfig::retention_days is singular
- **Notes:** Currently single-retention-window design. Per-label retention (Person kept 30 days, Package kept 3 days) is a natural Phase-2 extension. No test gap in Phase 1, but enumerated here as a clear future vector.
