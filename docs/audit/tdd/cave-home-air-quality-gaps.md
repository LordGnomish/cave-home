## Missing behaviors

### `air-quality-agg-001` — phase-1b — ✅ TDD-ready

AirGradient local HTTP API adapter: poll /measures endpoint and map pollutant fields onto Reading model

- **Spec:** ADR-019 Phase 1b; AirGradient's public API documentation (CO2, PM2.5, VOC-index, temperature, humidity)
- **Notes:** Strictly read-only HTTP poller; no authentication required for local-network endpoint. Maps wire format onto existing reading::Reading types, then feeds the already-tested engine. Network-bound I/O only.

### `air-quality-agg-002` — phase-1b — ✅ TDD-ready

Awair local API adapter: poll /air-data endpoint and map pollutant fields onto Reading model

- **Spec:** ADR-019 Phase 1b; Awair Element local API specification (CO2, PM2.5, PM10, NO2, VOC, temperature, humidity)
- **Notes:** Local-network variant only (no cloud auth required for Phase 1b per ADR-019). Strictly read-only poller; maps wire format to reading::Reading and reuses engine. Cloud variant deferred to Phase 2.

### `air-quality-agg-003` — phase-1b — ✅ TDD-ready

IKEA Vindriktning PM2.5 adapter: ingest UART sensor frame or Zigbee Z2M MQTT message, extract PM2.5 concentration, emit Reading

- **Spec:** ADR-019 Phase 1b; IKEA Vindriktning official sensor protocol and Zigbee2MQTT integration spec
- **Notes:** Hardware I/O bridge (UART via ESPHome or Zigbee via Z2M); single-pollutant sensor. Protocol parsing is straightforward; reuses engine. Hardware-dependent test fixture deferred.

### `air-quality-agg-004` — phase-1b — ✅ TDD-ready

Airthings radon adapter: ingest BLE advertisement frames or cloud API JSON, extract radon in Bq/m³, emit Reading

- **Spec:** ADR-019 Phase 1b; Airthings Wave/View sensor specification and Bluetooth LE manufacturer data format
- **Notes:** BLE for Wave (local), cloud API for View (account-bound). Phase 1b covers BLE path only per ADR-019. Radon is the sole pollutant; reuses engine. Cloud variant deferred to Phase 2.

### `air-quality-agg-005` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

cave-home-core state integration: expose RoomAssessment as a core State entity with overall category, dominant pollutant, and per-pollutant grades as sub-attributes

- **Spec:** ADR-019 Phase 1b; cave-home-core entity model (awaiting API stabilization). Charter §6.3 UX requirements for attribute exposure.
- **Notes:** Blocked by cave-home-core entity API not yet stable. Design is straightforward (RoomAssessment → entity); no new air-quality logic. Already implemented in core-agnostic crate; glue layer required once core API settles.

### `air-quality-agg-006` — phase-2 — ✅ TDD-ready

Multi-pollutant worst-of tie-breaking when two pollutants map to the same AirCategory: deterministic ordering (e.g., PM2.5 > CO2 > O3)

- **Spec:** Charter §6.3 requires stable dominant-pollutant attribution; current impl uses first-wins in iteration order (unstable).
- **Notes:** Current code: grades.iter().fold(...,|worst,dom|) picks first dominant, not last, in data order. For UX stability (same room → same dominant pollutant advice) a spec-grounded tie-breaker needed. Behavioral test can be written now against a chosen ordering spec.

### `air-quality-agg-008` — phase-2 — ✅ TDD-ready

AQI sub-index extrema caching: for a room with stable readings, cache the numeric AQI value and skip recomputation on re-assess

- **Spec:** Performance optimization for high-frequency reassess (e.g., every 10s). EPA AQI formula is deterministic; cache hit on stable input.
- **Notes:** Not essential for Phase 1 MVP; test can be written against a stable-input cache-hit scenario. Optimization only; no functional change.

### `air-quality-agg-009` — phase-2 — ✅ TDD-ready

NowCast unavailability logging: when now_cast returns None, indicate whether due to insufficient-recent-data, all-missing, or all-NaN scenario

- **Spec:** EPA-454/B-24-002 defines the validity rule (≥2 of 3 recent hours). Callers (Portal, automations) need to distinguish a transient sensor hiccup (None) from a stale sensor (None → offline).
- **Notes:** Current now_cast() returns bare Option<f64>; no enum variant for the 'why unavailable' case. Strict-TDD can target a NowCastResult enum with Unavailable(Reason). Behavior needed for UX diagnostics.

### `air-quality-agg-010` — phase-2 — ✅ TDD-ready

Multiple-pollutant NowCast fusion: apply EPA NowCast weighting to PM2.5 + PM10 independently, then report both or a fused 'particulate-matter' index

- **Spec:** EPA-454/B-24-002 defines separate NowCast for PM2.5 and PM10; no mandate to fuse. Current crate applies NowCast per-reading; phase-1b vendor adapters emit both.
- **Notes:** Design decision deferred: should room assessment report two AQI sub-indices (one per PM variant) or a single PM band? Test can be written against a chosen fusion spec.

### `air-quality-agg-007` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

Stale reading detection: RoomAssessment should indicate age of most recent & oldest reading, or declare assessment 'stale' if newest reading > threshold (e.g., 1h)

- **Spec:** Charter §6.1 liveness requirement; common in real-time systems (e.g., MQTT last_will for sensor disconnect). No spec binding yet.
- **Notes:** Not yet in crate or parity manifest. Sensor failure scenario (no update for > 1h). Needs a 'reading timestamp' field in Reading model (currently absent). Defer to Phase 2 pending Charter update.
