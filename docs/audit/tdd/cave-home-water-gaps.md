**Summary.** cave-home-water Phase 1 MVP implements the irrigation decision engine (zone model, seasonal/weather adjustment, soil-moisture gating, rain-delay/window/disabled handling with explainable skip reasons, flow-fault detection, sequential multi-zone run planning) with full EN/DE/TR localization. All mapped behaviors from the parity manifest are tested. Missing behaviors are Phase 1b/Phase-2 deferred (vendor I/O adapters, live weather feeds, real scheduling triggers, concurrent multi-zone groups) plus spec-gap behaviors not yet added: run-plan history tracking, decision-history audit trail, pressure-weighted runtime adjustment, and zero-sensor fallback strategy.

## Missing behaviors

### `run-plan-history` — phase-1b — ✅ TDD-ready

Ability to query and audit the history of past watering decisions and run-plans executed for a zone (for diagnostics, water-use trending, and accountability). The engine should track each decision made and the resulting plan, with timestamps.

- **Spec:** Home Assistant irrigation domain: decision audit trail; OpenSprinkler web UI run history and diagnostics.
- **Notes:** Currently decisions are computed but not persisted or queryable. A strict-TDD test can require a RunHistory trait/struct to store per-zone decisions with timestamps, and an API to retrieve them by date-range. Requires integration with a persistence layer (Phase 1b).

### `decision-history-audit` — phase-1b — ✅ TDD-ready

Log every watering decision decision (water / skip with reason) for a zone with the input state snapshot (soil moisture, seasonal %, rain delay, window, zone state) for post-hoc analysis and debugging. The engine should emit or store this audit trail for each decide() call.

- **Spec:** Home Assistant irrigation: decision log; OpenSprinkler: controller event log; Charter §2 accountability persona: 'I can see why my garden was/wasn't watered'.
- **Notes:** Currently decide() returns only a WaterDecision. An audit variant could return (WaterDecision, DecisionAudit) with full input context. Tests can verify that every decision captures soil moisture, seasonal %, rain delay, zone state, window flag, and the priority-order skip path taken.

### `fallback-watering-no-sensor` — phase-1b — ✅ TDD-ready

When a zone has no soil-moisture sensor (threshold=None), automatically fall back to a time-based watering schedule (e.g., every N days or only on specific days-of-week) instead of skipping the decision. Household should see 'Watering on schedule — no sensor' not 'Cannot decide'.

- **Spec:** Home Assistant irrigation: unsupervised (no-sensor) watering mode; OpenSprinkler: simple schedule mode (no sensor feedback) alongside smart mode. Charter §6.3: explainable fallback.
- **Notes:** Currently if there is no threshold, the moisture gate doesn't fire and the zone waters on other conditions. But there's no explicit 'nosensor' skip reason or fallback strategy. A test should verify that a zone with threshold=None and no sensor still gets an explicit fallback reason (not just 'oh, moisture gate didn't apply'). Requires a new SkipReason variant or a fallback_strategy field on Zone.

### `seasonal-percent-bounds-validation` — phase-1b — ✅ TDD-ready

Validate and clamp seasonal_percent to a reasonable range (e.g., 0..=200%) with an optional warning when out-of-range percentages are used. Currently accepts any u32, which can silently overflow or produce absurd runtimes.

- **Spec:** Home Assistant weather/seasonal integration: weather percent bounds; OpenSprinkler: 'watering time %' 0..200 slider UI. Best practice: prevent invalid inputs from silently producing bad decisions.
- **Notes:** Currently apply_seasonal_adjust() accepts any u32 and saturates on overflow (correct behavior). But there's no validation of the input or warning to the caller when seasonal_percent > 200. A test should verify clamping or explicit-bounds-check with a new error type. Already implemented for saturation (test-after only); bounds validation is missing.

### `window-conflict-with-disabled` — phase-1b — ✅ TDD-ready

When a zone is Disabled, the engine should explicitly report ZoneDisabled as the reason even if the moment is also outside the window. Currently does this correctly via priority order, but the spec should clarify: Disabled zone outside window → report Disabled, never OutsideWindow.

- **Spec:** Home Assistant irrigation: skip-reason priority (disabled > window); OpenSprinkler: disabled state overrides schedule.
- **Notes:** Already correctly implemented (test confirms priority: disabled beats window). This is a spec-clarification gap only — test-after: verify that a Disabled zone outside the window still reports ZoneDisabled, not OutsideWindow. Currently has a test (disabled_beats_rain_delay_in_priority) but no dedicated window test.

### `flow-sensor-required-runtime-minimum` — phase-1b — ✅ TDD-ready

For zones with a flow sensor, enforce a minimum runtime (e.g., >= 10 seconds) to allow the flow-fault detector to take a reliable reading. Short runs may produce noisy or inconclusive flow data. The engine should warn or reject runtimes below the sensor's minimum-reliable window.

- **Spec:** Flow-sensor best practice: reliable detection requires ≥N seconds of stable flow; Home Assistant sensor integrations: measurement stabilization time; OpenSprinkler: station-runtime constraints (min 10s for controller responsiveness).
- **Notes:** Currently Zone allows any runtime >= 1 second and detect() accepts any measured/expected flow. A test should verify that if a zone has a flow sensor (expected_flow_lpm=Some), its runtime is >= a minimum (e.g., 10s). Can be added as a Zone::new() validation rule. Pure TDD: fail test first requiring min-runtime enforcement for sensor-equipped zones.

### `paused-state-resume-priority` — phase-1b — ✅ TDD-ready

When a zone is Paused, the engine should allow resuming it (can_start=true) but track that it's a resumption, not a fresh start. Semantically: a paused zone may need to run longer to finish its original cycle, or it may skip the remainder if conditions change. The engine should expose resume-vs-start distinction in the decision.

- **Spec:** Home Assistant valve/irrigation: Paused state; OpenSprinkler: manual pause/resume for mid-cycle interruption recovery.
- **Notes:** Currently ZoneState::Paused.can_start() = true, and decide() treats it the same as Idle. A test should verify that a Paused zone gets a specific reason or flag in WaterDecision (e.g., 'Resuming paused watering' vs 'Starting new watering'). Requires extending WaterDecision with a resume_vs_start field or new SkipReason. Pure TDD: add a test expecting a PausedResume distinction that's not yet in the code.

### `rain-delay-duration-tracking` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Track how long a rain-delay has been active and automatically lift it after a configurable duration (e.g., 'rain delay for 12 hours after rain detected'). The rain_delay boolean input should be augmented with an optional duration so the engine can decide whether to keep delaying.

- **Spec:** Home Assistant rainfall + rain-delay automations: time-based rain-delay lift; OpenSprinkler: 'rain delay until' timer with automatic expiry. Smart water management: don't hold off forever.
- **Notes:** Currently decide() takes rain_delayed: bool (stateless). Requires clock integration (Phase 1b). A test could verify that given a rain-delay-started timestamp and a duration, the engine correctly lifts it after the window expires. Implementation deferred until clock/time integration lands.

### `pressure-weighted-runtime` — phase-2 — ✅ TDD-ready

When multiple zones can run in a watering cycle, adjust each zone's runtime inversely by the zone count so total water volume stays constant (pressure/supply-aware scheduling). A 3-zone cycle at 100% = each zone gets 66% time to preserve flow. Phase 2 feature underlying pressure-managed concurrent groups.

- **Spec:** OpenSprinkler multi-station model: pressure compensation via per-station cycle-time adjustment for grouped stations; Home Assistant irrigation: run-cycle pressure management.
- **Notes:** Not yet in the crate. A pure function could be added to adjust runtimes given a zone count: adjust_pressure_weighted(base_runtime, total_zones) -> adjusted_runtime. Already implemented (test-after only) for sequential runs (no pressure loss); pressure-weighted version is missing.

### `multi-zone-conflict-detection` — phase-2 — ✅ TDD-ready

Detect and report when a run-plan would exceed the household's water-supply capacity (e.g., 'running 2 zones at 15 lpm each = 30 lpm, but supply can only provide 20 lpm'). The engine should flag this as a warning or constraint violation before starting the plan.

- **Spec:** Home Assistant irrigation + concurrent multi-zone: pressure/flow-rate constraints; OpenSprinkler: 'max parallel stations' capacity checking.
- **Notes:** Currently plan_run() produces a sequential plan with no knowledge of supply-side constraints. A function plan_run_with_capacity_check(zones, decisions, max_lpm: f64) could verify that the max concurrent flow doesn't exceed supply. Tests can verify that overlapping zones (Phase 2 feature) are flagged when they'd exceed capacity.
