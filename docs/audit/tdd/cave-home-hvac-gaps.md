**Summary.** cave-home-hvac is a Rust port of Home Assistant's climate domain decision engine (generic thermostat hysteresis, HVAC modes, fan/preset modes, temperature conversion, capability gating, and EN/DE/TR UX labels). All 39 tests cover the implemented Phase-1 engine; missing behaviors are deferred vendor adapters (Phase 1b: Open3EClient, Daikin, LG, Bosch, Mitsubishi, Samsung), scheduling/PID, and core-entity integration (ADR-012). The engine itself is feature-complete per spec, but lacks stateful hysteresis tracking and humidity-based dehumidification control — behaviors that exist in Home Assistant but are not yet spec-validated for cave-home's use case.

## Missing behaviors

### `stateful-hysteresis-heat` — phase-1b — ✅ TDD-ready

Stateful hysteresis for Heat mode: maintain prior heating state across dead-band crossings so the system does not flip-flop between heating and idle within the tolerance band

- **Spec:** HA generic_thermostat — stateful hysteresis prevents relay chattering when current temperature oscillates within the dead-band; the system must 'remember' it was heating and keep heating until the upper threshold is crossed
- **Notes:** Current decide() is stateless; heat_decision() returns Idle when inside dead-band (line 96), ignoring prior state. HA maintains an internal 'last_action' to avoid oscillation. TDD-ready: can write a test that passes prior action and expects stateful retention.

### `stateful-hysteresis-cool` — phase-1b — ✅ TDD-ready

Stateful hysteresis for Cool mode: maintain prior cooling state across dead-band crossings to prevent rapid on/off cycles

- **Spec:** HA generic_thermostat — cool_decision must track prior state; stopping cooling only when room falls below target − cold_tolerance, not just when any temperature drop occurs
- **Notes:** cool_decision() (line 101) returns Idle for all temperatures below hot threshold, ignoring whether it was previously cooling. Needs to track and maintain cooling state within dead-band.

### `humidity-dehumidification-target` — phase-1b — ✅ TDD-ready

Dry mode should support both temperature-based (current) and humidity-based dehumidification targets, with current_humidity and target_humidity attributes to gate dehumidification when actual humidity exceeds target

- **Spec:** HA climate domain — Dry mode represents dehumidification; HA's climate entity supports current_humidity and target_humidity attributes (climate docs §dehumidification). Current implementation gates Dry only on temperature (line 140).
- **Notes:** Dry mode in decide() (line 137-144) compares room temperature to target + tolerance, but does not check actual humidity vs. target humidity. TDD-ready: can write test expecting humidity validation before Drying action is issued.

### `stateful-hysteresis-heatcool` — phase-1b — ✅ TDD-ready

HeatCool mode must track prior action (heating vs. cooling) to avoid flip-flopping when current temperature lands exactly between low + cold_tolerance and high + hot_tolerance

- **Spec:** HA generic_thermostat — HeatCool with hysteresis requires state memory; the system must cool if it was cooling and temp falls just inside the band, and heat if it was heating and temp rises just inside the band
- **Notes:** decide() for HeatCool (line 157-167) returns Idle for any temp between (low − cold_tol, high + hot_tol), ignoring prior heating/cooling state. Real thermostat exhibits hysteresis bias.

### `stateful-hysteresis-auto` — phase-1b — ✅ TDD-ready

Auto mode must maintain state to prevent chattering when temperature oscillates near the target setpoint within the dead-band

- **Spec:** HA generic_thermostat Auto mode — when current temp is within cold_tolerance below or hot_tolerance above target, Auto must hold the prior action (heating/cooling/idle) until a definitive boundary is crossed
- **Notes:** Auto decision (line 146-156) returns Idle for any temp inside [target − cold_tol, target + hot_tol], discarding prior state. TDD-ready: write test passing prior action and verifying it's held within dead-band.

### `off-mode-behavior-fan-stop` — phase-1b — ✅ TDD-ready

Off mode must guarantee the fan does not run; FanOnly mode is the only mode where the fan is guaranteed active even with no heating/cooling

- **Spec:** HA climate entity — HvacMode::Off must result in HvacAction::Off, which stops both heating/cooling AND fan. FanOnly runs the fan. These are mutually exclusive. Specification requires explicit validation.
- **Notes:** decide() returns HvacAction::Off for HvacMode::Off (line 127), which is correct; FanOnly returns Fan (line 128). However, no test explicitly validates that Off prevents fan, and no Capabilities validation prevents setting fan modes while in Off mode. TDD-ready: write test checking that fan is rejected in Off mode.

### `preheating-defrosting-transitions` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

HvacAction::Preheating and HvacAction::Defrosting must be explicitly entered/exited based on device state or time conditions; currently they are defined but never produced by decide()

- **Spec:** HA climate entity — HvacAction::Preheating (heat pump priming) and Defrosting (frost-melt cycle) are real actions. Home Assistant's generic thermostat does not emit these; vendor adapters (Viessmann, heat pumps) emit them based on device feedback. Specification requires state machine to manage transitions.
- **Notes:** decide() never returns Preheating or Defrosting (lines 127-167 cover all modes). These actions exist in HvacAction enum but are unreachable. Phase 1b vendor adapters will emit them based on device telemetry. TDD-ready only after device state machine is spec'd.

### `capabilities-humidity-validation` — phase-2 — ✅ TDD-ready

Capabilities should define min/max humidity limits and humidity step (granularity), with validation functions to reject out-of-range or off-step humidity targets, parallel to temperature validation

- **Spec:** HA climate entity — humidity_step, min_humidity, max_humidity are standard climate entity attributes (Apache-2.0 docs); home_assistant/core/components/climate/__init__.py defines these in ClimateEntityFeature and ATTR_* constants
- **Notes:** Capabilities struct (setpoint.rs line 86) only validates temperature range and step. No humidity bounds. TDD-ready: can write tests for check_humidity() analogous to check_temperature().

### `preset-mode-constraints-by-mode` — phase-2 — ✅ TDD-ready

Preset mode availability should validate that a preset is compatible with the requested HvacMode (e.g., Away preset may not be valid while in FanOnly mode)

- **Spec:** HA climate entity — preset_mode is mode-dependent; some presets (e.g., Away, Eco) are only sensible for heating/cooling modes, not FanOnly. Specification defines valid preset↔mode combinations per device/integration.
- **Notes:** Capabilities.check_preset() (line 235) does not validate mode context. Currently only checks device capability list. TDD-ready: can write tests for preset↔mode validation once spec-defined matrix is established.

### `tolerance-asymmetry-validation` — phase-2 — ✅ TDD-ready

Asymmetric cold vs. hot tolerances must be validated as sensible for the mode: some devices may have constraints (e.g., max asymmetry ratio) to prevent degenerate hysteresis bands

- **Spec:** HA generic_thermostat — cold_tolerance and hot_tolerance are independent but typically within the same order of magnitude. Extreme asymmetry (e.g., cold=10.0, hot=0.01) creates degenerate control behavior. Specification may define max-ratio constraints.
- **Notes:** Tolerance::new() (line 61) only checks that both are finite and non-negative; no validation of asymmetry ratio or domain-sensible bounds. TDD-ready: write test enforcing max asymmetry ratio (e.g., max_ratio = 10x).

### `fan-mode-constraints-by-hvac-action` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

Fan mode availability should be gated by current HvacAction: certain fan speeds (e.g., Low, Medium) may only be valid while actively heating/cooling, not while idle or off

- **Spec:** HA climate entity — supported_features and fan modes are context-dependent; not all devices support all fan speeds in all modes. Fan constraints are checked against current action, not just device capabilities.
- **Notes:** Capabilities.check_fan_mode() (line 222) only checks if the device lists the fan mode; does not consider current HvacAction. This is deferred to Phase 2 and requires stateful context (prior action).

### `setpoint-temperature-precision-step` — permanent — ✅ TDD-ready

Temperature setpoint precision (target_step) should be validated to ensure it is not finer than the device's temperature sensor precision, to avoid requesting unachievable targets

- **Spec:** HA climate entity — target_step defines the granularity of temperature changes the device accepts. Requesting a target that falls on the step is essential; requesting a target finer than the sensor can measure is wasteful. Specification requires step validation.
- **Notes:** Capabilities.check_temperature() (line 188) validates that temp lands on the target_step, which is correct. However, no validation that the step itself is reasonable relative to sensor precision. TDD-ready: write test checking that target_step is within domain bounds (e.g., >= 0.1 °C).
