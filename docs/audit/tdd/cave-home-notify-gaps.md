- **Upstream:** none (first-party ntfy-class semantics)

- **Existing cave tests (agent count):** 40
- **Missing behaviors identified:** 12 (10 strict-TDD-ready)

> Generated by the spec-based gap-analysis pass (read-only). For
> `gpl-agpl-cleanroom` upstreams no upstream source/tests were consulted —
> behaviors are derived from public protocol/spec knowledge only.

**Summary.** cave-home-notify is a well-tested first-party notification routing engine (40 tests covering 6 core modules). The crate implements the Phase-1 MVP of ADR-021: priority scale, topic validation, notification model, subscriber routing, dedup cache, rate limiter, and i18n rendering — all pure, std-only logic with no clock or transport dependencies. Existing tests are spec-driven and achieve strong coverage of the main happy paths and boundary conditions. Missing behaviors fall into three categories: (1) Phase-1b deferred items (ClickAction rendering, attachment transport capabilities) that are untestable until transports ship; (2) pure logic edge cases and second-order effects (builder chaining order, em-dash encoding, fractional refill accumulation) that are correct but underspecified in tests; and (3) performance/scale concerns (10k subscriber routing, large dedup key volume). The crate is ready for Phase 1 and has no spec-grounded gaps in its stated scope.

## Missing behaviors

### `notif-click-action-types` — phase-1b — ✅ TDD-ready

ClickAction enum supports OpenView and OpenLink variants with distinct semantics; render tests verify the model carries these, but no test validates that a transport layer can distinguish between view-local navigation (OpenView) and external URLs (OpenLink) in the DeliveryPlan context

- **Spec:** src/notification.rs::ClickAction; cave-home Notification model design (ADR-021): click-action metadata is transport-agnostic in Phase 1
- **Notes:** ClickAction rendering (what a household sees) is untested; only the model structure is tested. Phase-1b transports will interpret these differently. A strict TDD test would verify that both variants exist and are distinguishable in a Notification.

### `attachment-metadata-handling` — phase-1b — ✅ TDD-ready

Attachment struct carries name and optional size_bytes; the Notification model includes it, but no test validates that attachment metadata survives routing decisions (DeliveryPlan) or is correctly handled when a notification with an attachment is routed to different channels with different attachment capabilities

- **Spec:** src/notification.rs::Attachment; ADR-021 Phase 1b: attachment bytes fetch/stream deferred; model carries metadata only
- **Notes:** Model structure tested but attachment behavior in routing context untested. Phase-1b transports will need to validate whether a channel (e.g., SMS) can actually receive attachments. TDD test: route a notification with an attachment to a subscriber with mixed channels and verify the Delivery records are created regardless (Phase-1b transport decides capability).

### `topic-display-impl` — phase-2 — ✅ TDD-ready

Topic implements core::fmt::Display via a fmt impl; the trait is public and satisfies Rust conventions, but no test explicitly validates that Display output matches as_str() and formats correctly in string contexts

- **Spec:** src/topic.rs::impl Display for Topic
- **Notes:** Trait implementation is present but untested. A test would call format!("{}", topic) and verify it equals topic.as_str().

### `rate-limiter-overflow-handling` — phase-2 — ✅ TDD-ready

RateLimiter uses u32::try_from(gained).unwrap_or(u32::MAX) to handle u64 overflow when refilling many tokens; the logic saturates and clamps, but no test validates the behavior when a very large time gap (where gained > u32::MAX) occurs

- **Spec:** src/throttle.rs::RateLimiter::admit line 132; token bucket algorithm edge case: refill overflow handling
- **Notes:** The saturating add and min(burst) clamp ensure safety, but the conversion to u32 with fallback is not explicitly tested. TDD test: admit a notification, let u64::MAX ticks pass, verify tokens are clamped to burst (not saturated to u32::MAX).

### `dedup-cache-memory-pressure` — phase-2 — ✅ TDD-ready

DedupCache::prune removes entries older than window ticks; no test validates the memory behavior under realistic high-volume notification scenarios (e.g., many unique dedup keys over time, then pruning reduces memory as expected)

- **Spec:** src/throttle.rs::DedupCache::prune; cache maintenance and memory lifecycle
- **Notes:** Functional correctness (prune drops old entries) is tested, but practical memory impact is not. TDD test: create a cache, admit 1000 unique notifications, prune, verify tracked() count matches expected subset.

### `channel-all-const-coverage` — phase-2 — ✅ TDD-ready

Channel::ALL constant provides all four channel variants for iteration; no test validates that ALL contains exactly the four enum variants and no duplicates, or that exhaustive matching against ALL never misses a variant

- **Spec:** src/route.rs::Channel::ALL constant
- **Notes:** The constant is defined but not explicitly tested for completeness. TDD test: verify length, verify no duplicates, verify enum_iterator (if used) matches ALL count.

### `label-coverage-all-languages` — phase-2 — ✅ TDD-ready

Priority::label returns a static &str for all 5 priorities and all 3 languages (15 combinations); existing tests verify they are non-empty and distinct per priority, but no test validates every single label string (e.g., for typos, accents, or translation correctness beyond 'not empty')

- **Spec:** src/label.rs::Priority::label; Charter §6.3 EN/DE/TR mandatory languages from M1
- **Notes:** A comprehensive test matching each (Priority, Lang) pair to a golden expected label string would verify translation accuracy and prevent regressions (e.g., Unicode normalization issues).

### `notification-builder-chaining-order` — phase-2 — ✅ TDD-ready

Notification builders (with_priority, with_tags, with_click, with_attachment) can be chained in any order; the final state should be independent of order, but no test validates commutativity or that all builder methods properly mutate the same struct

- **Spec:** src/notification.rs::Notification builder methods; builder pattern design
- **Notes:** Order-independence is implied by the struct design but not tested. TDD test: build the same notification with builders in different orders and verify equality.

### `render-format-consistency` — phase-2 — ✅ TDD-ready

Notification::render produces exactly one format with separator ' — ' (em-dash) between title and body; no test validates that the em-dash character is correctly preserved across different encodings or platforms, or that render output is consistent with i18n label output

- **Spec:** src/label.rs::Notification::render format: '{label}: {title} — {body}'; Charter §6.3 UX consistency
- **Notes:** Format is tested but em-dash encoding (UTF-8 U+2014) is not explicitly validated. TDD test: verify render output includes the exact em-dash character and handles titles/bodies with Unicode correctly.

### `rate-limiter-fractional-interval` — phase-2 — ✅ TDD-ready

RateLimiter::admit tracks fractional time ('leftover fractional time still counts toward the next token') by advancing last_refill only by consumed intervals; no test validates the cumulative effect over many fractional refills (e.g., refill_interval=3, gaining 0.33 tokens per tick should accumulate to one token after ~9 ticks)

- **Spec:** src/throttle.rs::RateLimiter::admit lines 130-136; token bucket: fractional refill accumulation
- **Notes:** Correct implementation but the fractional accumulation behavior is not explicitly tested. TDD test: admit with refill_interval=3, drive ticks forward in non-aligned steps (e.g., +1, +1, +1), verify one token refills at the third step.

### `subscriber-topic-matching-edge-case` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

Subscriber::accepts checks topic equality via == and priority via meets; the behavior is correct but no test validates the case where two topics have identical names (after Topic validation) result in equal subscribers, or validates behavior with a very large number of subscribers and cross-topic filtering efficiency

- **Spec:** src/route.rs::Subscriber::accepts; routing logic: topic matching and filtering
- **Notes:** Already implemented and tested; test-after only. The core logic is sound, but performance/scale testing (e.g., route() with 10k subscribers) is not covered.

### `topic-dedup-in-delivery-plan` — permanent — ⏸ not-yet (needs impl-deps / test-after)

DeliveryPlan::recipients() deduplicates by subscriber name, preserving first-seen order; it does not deduplicate by topic. No test validates the case where a single subscriber receives the same notification over multiple channels but from different routing paths (e.g., overlapping topic subscriptions, if that were possible)

- **Spec:** src/route.rs::DeliveryPlan::recipients; routing decision: subscriber deduplication
- **Notes:** Already implemented and tested correctly (each Subscriber is single-topic); test-after only. The current design is that a Subscriber subscribes to one topic, so no overlapping path exists. This gap is a non-issue by design.
