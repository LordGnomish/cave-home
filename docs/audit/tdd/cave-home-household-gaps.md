**Summary.** cave-home-household is a pure, no-clock household management engine ported spec-first from Grocy (MIT). Phase 1 implements the decision engine: product inventory model, stock operations with overdraw protection, below-min shopping with purchase-unit rounding, expiry classification, recurring chore scheduling, recipe stock-checks, and localisation. Storage, REST API compatibility, barcode import, and calendar integration are Phase 1b. 35 tests cover the implemented behaviors; identified gaps focus on real-date support, constraint checking, and advanced chore/recipe features acknowledged as deferred in ADR-026 and parity.manifest.toml."

## Missing behaviors

### `real_dates_support` — phase-1b — ✅ TDD-ready

Convert real calendar dates (ISO 8601 / timestamp) to caller-supplied integer day numbers for best-before, chore due dates, and recipe tracking

- **Spec:** ADR-026 'Real calendar-date best-before / due dates (timezone-aware)'; parity.manifest.toml [[unmapped]] 'the engine uses caller-supplied integer day numbers so it stays pure'
- **Notes:** The engine is deterministic because it takes 'today' as an integer. Converting real dates and timezones to day numbers is acknowledged as an 'adapter concern at the edge' (Phase 1b). A pure conversion module could land in Phase 1b to bridge the gap.

### `product_quantity_constraints` — phase-1b — ✅ TDD-ready

Validate and enforce quantity unit constraints: e.g. a product with Pack(6) cannot be partially consumed (must respect pack integrity)

- **Spec:** Grocy inventory model: quantity-unit-aware stock operations maintain physical integrity of packaged goods
- **Notes:** Currently stock::consume/purchase accept any float amount. A strict-packing mode would prevent 'consuming 2 from a six-pack' (should consume 6 or fail). Not mentioned in current source or tests; spec-grounded from Grocy's unit constraints.

### `shopping_list_priority` — phase-1b — ✅ TDD-ready

Mark shopping items with priority tiers (urgent, normal, nice-to-have) so the Portal can render and sort them by urgency

- **Spec:** Grocy shopping list: 'item category / priority' for organization; Grocy REST API includes status field (not done, done)
- **Notes:** ShoppingItem currently carries name, amount, unit, manual flag. No priority or category. A priority enum would let households distinguish 'milk (urgent: below minimum)' from 'optional nice-to-have'.

### `product_category_filter` — phase-1b — ✅ TDD-ready

Categorize products (food, medicine, batteries, cleaning supplies) so the Portal and shopping app can filter and organize by type

- **Spec:** Grocy product model: 'product category' field (food, non-food, etc.); Grocy inventory distinguishes 'food vs medicine vs batteries'
- **Notes:** Product model has no category field. Grocy distinguishes them for regulatory/display purposes. Adding a Category enum would allow Portal to show 'Medicines' tab vs 'Food' vs 'Batteries', aligning with Grocy's separation.

### `batch_stock_operations` — phase-1b — ✅ TDD-ready

Perform multiple stock operations atomically: 'consume 2 eggs and 1 cup milk for this recipe' as one transaction with rollback on partial failure

- **Spec:** Grocy stock operations: 'move stock out for recipe prep' (implicit transactionality); ADR-026 Phase 1b storage implies transaction support
- **Notes:** Current stock::consume takes one product at a time. A batch operation (consume_for_recipe) would ensure all-or-nothing when prepping a meal. Not yet in the MVP; spec-grounded from Grocy's use-case of simultaneous ingredient consumption.

### `opened_package_tracking` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Track opened packages separately from unopened inventory (opened packages may have shorter shelf-life rules or visibility marks)

- **Spec:** stock.rs::open 'Phase 1b store can record opened-package state'; Grocy inventory module distinguishes opened vs unopened stock
- **Notes:** open() currently is a no-op-on-stock that only validates a package exists. Phase 1b storage will need to distinguish opened state, but the MVP engine does not track it. Already implemented (as no-op), test-after only.

### `negative_stock_allowance` — phase-2 — ✅ TDD-ready

Support 'allow overdraw' flag per product: some items (e.g. spices bought in bulk) may legitimately go negative before restock

- **Spec:** Grocy inventory advanced settings: 'allow negative quantity' toggle per product
- **Notes:** Currently all stock operations reject overdraw as an error. A product-level flag would enable scenarios like 'I have spices worth tracking but don't reorder until stock hits -5 (used regularly)'. Not present in current Product or stock module.

### `chore_recurrence_patterns` — phase-2 — ✅ TDD-ready

Support complex recurrence patterns: not-on-weekends, skip-on-holiday, first-of-month, every-other-week (beyond simple period_days)

- **Spec:** Grocy chore scheduling: 'chore interval' (days, weeks, months, custom sequences); ADR-026 'recurring task scheduling'
- **Notes:** Currently Chore only stores period_days (simple daily interval). A full recurrence system (RRULE-like) would cover monthly chores ('take bins out first Monday'), holiday handling, etc. Spec-grounded from Grocy's 'chore interval' feature.

### `recipe_prep_time` — phase-2 — ✅ TDD-ready

Store and report recipe prep time, cooking time, and servings (beyond just ingredients and stock-check)

- **Spec:** Grocy recipe module: time_needed, servings, instructions, photos; Charter §6.3 'no clock' exempts prep-time reporting from being deferred
- **Notes:** Recipe currently holds name + ingredients only. Grocy's recipes track prep-time (e.g. '30 min prep + 1 hour bake'), servings count (e.g. 'serves 4'), and instructions. Serves and times are context (not pure stock-check logic) but valuable for UX.

### `expiry_custom_window` — phase-2 — ✅ TDD-ready

Support per-product expiry warning windows (e.g. milk warns 3 days before, medicine warns 30 days before)

- **Spec:** Grocy product settings: 'min_stock_warning_days' analogue for expiry (implicit: different product types have different safety windows)
- **Notes:** Freshness::classify takes a global 'within' window for all products. Grocy allows per-product safety margins. A Product field would let medicine use a wider window than dairy.

### `unit_conversion` — phase-2 — ✅ TDD-ready

Convert between compatible units (e.g. millilitres ↔ litres, grams ↔ kilograms) for shopping and recipe matching

- **Spec:** Grocy product model: implicit unit-conversion for flexible entry (e.g. buy 2 L milk when stored as ml); common recipe/shopping scenario
- **Notes:** QuantityUnit is an enum (Piece, Pack, Grams, Millilitres) with no conversion path. Recipes require exact unit match. Adding gl/litre, mg/g would enable '1 L milk in recipe, but we store 1000 ml'.

### `chore_history_count` — permanent — ⏸ not-yet (needs impl-deps / test-after)

Track how many times a chore has been completed (beyond just last_done date) for summary stats ('vacuumed 45 times this year')

- **Spec:** Grocy chore tracking: implicit completion count for reporting; home-automation analogy: entity history tracking
- **Notes:** Chore stores only last_done (Option<i64>), not a completion count. Phase 1b storage will append chore-complete events; the MVP engine only knows last done date. Not a spec gap — data-binding issue (deferred to store).
