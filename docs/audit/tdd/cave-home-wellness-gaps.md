- **License class:** permissive

> `gpl-agpl-cleanroom` upstreams no upstream source/tests were consulted —

**Summary.** cave-home-wellness is a privacy-first wellness decision engine (Apache-2.0 upstream, Home Assistant integrations) with 31 comprehensive tests covering metric validation, goal progress, band classification, trend analysis, and streak counting. The core engine is feature-complete and spec-grounded in public wellness guidance. Missing behaviors are primarily phase-1b/phase-2 deferrals (weekly goal aggregation, weight bands, edge-case handling) or quality refinements (NaN resilience, metric arithmetic, localization extensibility). All unmapped areas (wearable adapters, on-device history, cloud uploads) are ADR-025 justified and deferred; permanent boundary prevents cloud sync per Charter §9.

## Missing behaviors

### `body-weight-band` — phase-1b — ✅ TDD-ready

Weight-status wellness band (Underweight/Normal/Overweight/Obese) with localized EN/DE/TR names and gentle advice, integrating BodyWeight metric into band classification like the three existing bands (RestingHrBand, SleepBand, ActivityBand)

- **Spec:** General-population BMI reference ranges (WHO, CDC); parity.manifest.toml §mapped implies all metric types should have band classification
- **Notes:** BodyWeight value object exists (metric.rs) but no band classification exists. This is a straightforward port of HA wellness weight status bands once threshold spec is confirmed (currently missing from spec_sources list).

### `trend-with-nan-handling` — phase-1b — ✅ TDD-ready

Trend classification must reject or gracefully handle NaN/infinite values in input series (e.g., malformed sensor data), not panic or produce undefined results

- **Spec:** trend.rs::classify_trend processes user-supplied f64 series; no explicit NaN handling documented. BodyWeight::new() validates finiteness; trend classifier should match.
- **Notes:** classify_trend(&[f64]) has no guards against NaN/INFINITY in the series. Should either return Result or treat NaN as 0.0/skip. Current code will produce NaN results silently.

### `goal-progress-overflow-edge-cases` — phase-1b — ✅ TDD-ready

Goal progress calculation must handle extreme observed values without overflow or precision loss (e.g., observed=u32::MAX, target=1 should clamp to 100%, not overflow)

- **Spec:** goal.rs::progress uses u64 intermediate; no tests for boundary cases like u32::MAX inputs, or metric maximums (Steps::MAX=200k, SleepDuration::MAX_MINUTES=1440)
- **Notes:** Only 'over_target_clamps_percent' test exists; missing tests for metric-maximum inputs (e.g., goal 10k steps vs observed Steps::MAX, goal 480min sleep vs max). Current math is sound but edge-case coverage is incomplete.

### `weekly-goal-aggregation` — phase-2 — ✅ TDD-ready

Weekly goal progress aggregation: given a 7-day DailyMetrics series and a Goal with Period::Weekly, compute cumulative progress (sum for steps/active, average for sleep/HR)

- **Spec:** Goal struct has Period::Weekly variant; parity.manifest.toml goal-engine spec does not distinguish daily vs. weekly aggregation logic. ADR-025 mentions 'weekly activity balance' automations.
- **Notes:** Goal::progress() only handles single-day metrics. Weekly aggregation (step total over 7 days, avg sleep/HR, etc.) requires new pub fn weekly_progress(days: &[DailyMetrics], goal: Goal) -> GoalProgress logic.

### `streak-non-chronological-input` — phase-2 — ✅ TDD-ready

streak functions document that input series is assumed chronological; should validate or document behavior if days are out-of-order (e.g., day.day ordinals not monotonic)

- **Spec:** streak.rs::current_streak docstring says 'series assumed chronological (oldest first)'; no validation of DailyMetrics::day field ordering
- **Notes:** Empty test coverage for unsorted or reversed input. Either validate precondition or add guards. Currently relies on silent assumption that caller maintains order.

### `metric-arithmetic-tests` — phase-2 — ✅ TDD-ready

Comprehensive tests for metric-to-metric arithmetic (e.g., can accumulate SleepDuration::from_minutes(480) + SleepDuration::from_minutes(300), or compare across metric types), boundary conditions (0, MAX)

- **Spec:** metric.rs defines Steps(u32), HeartRate(u16), SleepDuration(u16), ActiveMinutes(u16), BodyWeight(f64) — each constructor validates bounds. No explicit operator tests (Add, Sub, Div for averaging/aggregation).
- **Notes:** Metrics are immutable value objects with no arithmetic ops defined. Likely by design (purity), but weekly aggregation (phase-2) will need addition/averaging; tests should validate that aggregated values stay within plausibility bounds.

### `band-boundary-rounding` — phase-2 — ✅ TDD-ready

Tests verifying that band classification thresholds are applied consistently under floating-point edge cases (e.g., SleepBand::from_minutes(419) vs (420), when callers provide fractional minutes rounded to u16)

- **Spec:** SleepBand::from_minutes uses u16 boundaries (420, 480, 540); band.rs tests cover exact boundaries. No tests for intermediate rounding or accumulation artifacts (e.g., 7 nights of 71.4-min sleep = 499.8 min total).
- **Notes:** Current tests check hardcoded boundary values; missing coverage for common rounding scenarios once weekly aggregation arrives.

### `localization-extensibility` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

Lang enum (En, De, Tr) is hardcoded; provide clear migration path or tests for adding new languages (e.g., ES, FR, RU) to band names/advice without breaking existing band-impl pattern-matches

- **Spec:** Charter §6.3 specifies EN+DE+TR from M1; label.rs::Lang is sealed 3-variant enum; band.rs::{name,advice} use exhaustive match. ADR-025 mentions TR explicitly; extensibility plan is not documented.
- **Notes:** Already implemented but lacks future-proofing. Adding a 4th language requires touching band.rs match arms; no tests validate that all band+lang combinations exist. Consider marker test for completeness.

### `multilingual-advice-consistency` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

Tests ensuring that band names and advice across EN/DE/TR maintain semantic equivalence and non-clinical tone, not just character-set correctness (e.g., German 'etwas' vs English 'a little' convey the same nuance)

- **Spec:** band.rs::ui_strings_carry_no_implementation_jargon and ui_strings_carry_no_clinical_diagnosis_words tests check for banned terms; no semantic equivalence tests across languages.
- **Notes:** Bans clinical/jargon terms per Charter §6.3; missing is verification that non-English translations maintain the original gentle, non-alarming tone. Human review / subject-matter expert validation needed, not TDD-ready.

### `daily-metrics-field-validation` — deferred — ✅ TDD-ready

DailyMetrics::new() accepts pre-validated metric objects without validation; document or test that day ordinal is sensible (e.g., no negative, no giant jumps that suggest data corruption)

- **Spec:** DailyMetrics::day is u32 opaque ordinal; used only for ordering and streaks per lib.rs docstring. No bounds or semantic validation.
- **Notes:** Currently any u32 is accepted. For on-device history (phase-1b), should validate that day ordinals are monotonically increasing or at least in a sane range. Low priority; mostly documentation.
