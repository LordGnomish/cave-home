# TDD gap analysis — `cave-home-lock`

- **Upstream:** home-assistant/core (lock-domain integrations, Apache-2.0) + Nuki/SwitchBot public REST APIs
- **License class:** permissive
- **Port method:** spec-based (lock state machine + Home Assistant lock-domain semantics) + first-party (safety/PIN logic)
- **Existing cave tests (agent count):** 40
- **Missing behaviors identified:** 11 (11 strict-TDD-ready)

> Generated by the spec-based gap-analysis pass (read-only). For
> `gpl-agpl-cleanroom` upstreams no upstream source/tests were consulted —
> behaviors are derived from public protocol/spec knowledge only.

**Summary.** cave-home-lock is a permissive-licensed (Apache-2.0) port of Home Assistant's lock-domain integrations, implementing the lock state machine, keypad PIN verification, and localized UI labels. With 40 existing tests, the crate covers core state transitions, capability gating, jam handling, and brute-force lock-out. Missing behaviors cluster around: (1) exhaustive direction-reversal / abort scenarios, (2) Open command edge cases (jam during Open, transitioning from Open, keypad-gated Open), (3) parameterized boundary testing for PIN length and failure-count saturation, and (4) comprehensive credential lock-out persistence. All identified gaps are TDD-ready against the public Home Assistant lock-domain spec.

## Missing behaviors

### `abort-lock-direction-reversal` — phase-1b — ✅ TDD-ready

User-initiated command abort: issuing the opposite command (Unlock) while a lock is mid-Locking should successfully reverse direction and enter Unlocking state, and vice versa. This behavior is marked as legal in the transition matrix but lacks explicit integration test coverage demonstrating the full abort workflow.

- **Spec:** ADR-016 lock state machine: 'the lock then *confirms* (settles to Locked / Unlocked / Open) or *fails* (settles to Jammed)'; Home Assistant lock domain: direction-reversal semantics are part of optimistic updates.
- **Notes:** Transition rule exists (machine.rs line 280-281) and test exists (opposite_command_can_abort_in_flight), but only covers single direction reversal. No test covers: start Locking → reverse to Unlocking → confirm → settled to Unlocked scenario.

### `lock-from-open-state` — phase-1b — ✅ TDD-ready

A lock in Open state (latch retracted) should be able to receive Lock or Unlock commands and transition normally. The current tests do not cover applying Lock/Unlock commands from the Open state to verify the state machine allows these transitions and settles correctly.

- **Spec:** Home Assistant lock domain: Open is a valid lock state; machine.rs is_legal() permits transitions from any non-jammed state except redundant in-flight commands.
- **Notes:** The is_legal() method should allow Open → Locking or Open → Unlocking, but no explicit test verifies 'lock applied from Open state goes to Locking then Locked'.

### `clear-jam-with-open-observed-state` — phase-1b — ✅ TDD-ready

After a jam, clear_jam() allows the user to report the observed (now-known) state. If the lock has Open capability and the observed state is Open (e.g., the human found the door already open), clear_jam(LockState::Open) should succeed and transition the lock to Open. Currently, tests clear_jam to Locked or Unlocked, but not to Open.

- **Spec:** machine.rs line 188-197: clear_jam validates observed state is not transient or jammed; ADR-016: Open is a settled state.
- **Notes:** clear_jam_to_settled_state and clear_jam_rejects_transient_observation test with Locked/Unlocked, but not with Open even though it's a valid settled state.

### `transient-state-after-code-rejected` — phase-1b — ✅ TDD-ready

When apply_with_code is called with a wrong PIN, the lock must not change state (remain in its pre-command state). The tests verify this for Locked → Unlock case, but not exhaustively for all state+command+wrong-code combinations (e.g., Unlocked → Lock with wrong code, or Unknown → Unlock with wrong code).

- **Spec:** machine.rs line 156: 'CodeVerdict::Rejected | CodeVerdict::LockedOut => Err(TransitionError::CodeRejected)'; ADR-016: 'code rejection must not move the lock'.
- **Notes:** Test keypad_unlock_with_wrong_code_rejected exists, but only for Locked → Unlock. No test for other state/command pairs with wrong code.

### `lockout-persists-across-correct-attempts-in-cascade` — phase-1b — ✅ TDD-ready

After a credential is locked out, issuing multiple consecutive attempts (including correct PINs) should all return LockedOut until reset() is called. The test locked_out_refuses_even_correct_pin verifies one scenario; an exhaustive sequence test ensures the lock-out is 'sticky' and does not accidentally clear on subsequent attempts.

- **Spec:** code.rs line 228-230: 'if self.is_locked_out() { return CodeVerdict::LockedOut; }'; semantics: lock-out is terminal until reset.
- **Notes:** Test locked_out_refuses_even_correct_pin exists but only checks one correct attempt after lock-out. No test for 'locked out → attempt N times → all return LockedOut'.

### `code-length-boundary-at-max-12-plus-one` — phase-1b — ✅ TDD-ready

A PIN of 12 digits should be accepted (boundary case); a PIN of 13 digits should be rejected with TooLong. The test rejects_too_long covers this, but there's no exhaustive boundary test verifying MIN_CODE_LEN (4) is the absolute floor and MAX_CODE_LEN (12) is the absolute ceiling with no off-by-one errors.

- **Spec:** code.rs: MIN_CODE_LEN = 4, MAX_CODE_LEN = 12; parse() enforces 'raw.len() < MIN_CODE_LEN' and 'raw.len() > MAX_CODE_LEN'.
- **Notes:** Tests cover boundary cases individually (accepts_minimum_length, accepts_maximum_length, rejects_too_long), but no single parameterized test verifies all adjacent boundaries (3,4,5... 11,12,13).

### `jammed-lock-clears-only-via-clear-jam` — phase-1b — ✅ TDD-ready

A lock in Jammed state must only transition out via clear_jam(), never by apply() or confirm() or fail(). The tests command_on_jammed_lock_are_rejected verify apply() is rejected, but there's no explicit test for confirm() or fail() being called on a Jammed lock and confirming they are idempotent (do not clear the jam).

- **Spec:** machine.rs line 104-105: Jammed locks reject all apply() commands; line 166-172 confirm() is idempotent; line 178-180 fail() on any state sets Jammed.
- **Notes:** confirm() called on Jammed lock is tested implicitly (confirm_is_idempotent_from_settled covers this), but there's no explicit test 'confirm_on_jammed_stays_jammed' for clarity.

### `jam-from-open-command` — phase-2 — ✅ TDD-ready

When a lock with Open capability receives an Open command, it should settle directly to Open state (no in-flight phase). However, if the hardware reports failure/jam during the Open action, the lock should transition to Jammed. Currently, Open has no distinct transient state; no test covers hardware jam reported after an Open command is sent.

- **Spec:** Home Assistant lock domain: Open command maps to 'retract latch so the door can be opened'; ADR-016: 'jam is safety-critical'; machine.rs line 79-84 shows Open has no in_flight_state.
- **Notes:** Open settles immediately (line 136); test open_supported_settles_to_open confirms this. But there's no test for 'what if fail() is called after applying Open' — the fail() method would transition from Open to Jammed, but this scenario is not explicitly tested.

### `keypad-open-command-with-code` — phase-2 — ✅ TDD-ready

A lock with Open capability should support apply_with_code(LockCommand::Open, ...) to gate the Open command behind PIN verification. Currently, tests cover keypad-gated Lock and Unlock, but not keypad-gated Open.

- **Spec:** machine.rs apply_with_code() handles all three LockCommand variants; no special case excludes Open.
- **Notes:** The apply_with_code method would work for Open (it calls apply() after credential check), but there's no explicit test case: 'keypad_open_with_correct_code' or 'keypad_open_unsupported_on_deadbolt'.

### `credential-verify-saturating-failure-count` — permanent — ✅ TDD-ready

The failure counter uses saturating_add (line 235) to prevent overflow; this is correct, but there is no explicit test verifying that once failure_count reaches u32::MAX or is very large (e.g., after 1000 failed attempts), subsequent failures still behave correctly and do not panic or wrap.

- **Spec:** code.rs line 235: 'self.failures = self.failures.saturating_add(1)'; Rust safety contract: saturation must not cause UB.
- **Notes:** Edge case: theoretical but important for robustness. A lock-out counter that saturates is better than one that overflows, but no test verifies the saturation boundary.

### `state-label-and-advice-for-all-language-state-pairs` — permanent — ✅ TDD-ready

Every combination of (LockState, Lang) must have a label and advice, and both must be non-empty, non-vendor-jargon. The test every_state_has_three_language_label_and_advice verifies coverage, but a parameterized exhaustive test ensures no state/language pair is missing.

- **Spec:** Charter §6.3: EN/DE/TR required; ADR-007: labels must be household-friendly; label.rs line 22-46, advice line 50-74.
- **Notes:** Test every_state_has_three_language_label_and_advice is comprehensive, but it's a single loop test. A parameterized #[test] for each (state, lang) pair would catch missing cases at compile time.
