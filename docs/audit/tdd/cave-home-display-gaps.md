**Summary.** cave-home-display is a Phase 1 MVP control engine for TV/display integration (LG webOS, Samsung Tizen, Android TV/Cast, Portal wall-panel). Implements power states, playback transitions, volume clamping with parental cap, source/app catalog validation, command gating, and status localization (EN/DE/TR). 44 existing tests cover all 6 primary modules. Missing behaviors are edge cases in state transitions, multi-app scenarios, and specification clarifications for Home Assistant media_player semantics that are not yet tested.

## Missing behaviors

### `app-replacement-transition` — phase-1b — ✅ TDD-ready

Launching a new app while another app is already running should cleanly replace the previous app without requiring SelectSource first. Currently there is no test or defined behavior for app-to-app transitions.

- **Spec:** Home Assistant media_player: LaunchApp service implies replacing the current playback app. HA spec: only one app can be 'active' at a time; launching differs from SelectSource.
- **Notes:** Behavior is likely correct (current_app would be overwritten), but no explicit test. Should verify: Launch Netflix, Launch YouTube, current_app == YouTube, position resets.

### `idle-to-playing-buffering-flow` — phase-1b — ✅ TDD-ready

When play() is called on an Idle display (home screen with nothing loaded), the spec is silent on whether it should transition to Playing (ready to play once content is loaded) or Buffering (waiting for content). Currently, play() always goes to Playing, even from Idle, which may be semantically incorrect.

- **Spec:** Home Assistant media_player PlaybackState: Playing = content is playing; Idle = on but nothing loaded. Play on Idle should not immediately be Playing (nothing to play yet). Typical behavior: Idle → Buffering → Playing.
- **Notes:** Current code: PlaybackState::Idle.play() == PlaybackState::Playing. Likely incorrect. Test: powered_on TV (Idle), apply Play, should be Buffering or error (nothing to play). Then LaunchApp, apply Play, should be Playing. Requires changing playback.rs::play() to handle Idle specially.

### `standby-resume-state-preservation` — phase-2 — ✅ TDD-ready

When powering on from Standby (vs Off), the TV should resume its previous app/source context rather than resetting to Idle. Currently, PowerOn always resets playback to Idle regardless of prior state.

- **Spec:** Home Assistant media_player entity: wakeup semantics distinguish fast-resume (Standby→On preserves context) from cold-boot (Off→On resets). ADR-028 §Standby definition: 'listening for wake'.
- **Notes:** Currently On state is achieved but app/source are lost. Test should: power on, launch app, force to Standby, power on again, verify app is still active. Requires adding previous_app/previous_source fields to track pre-standby state.

### `seek-on-buffering-state` — phase-2 — ✅ TDD-ready

Seeking while the player is in Buffering state should either be an error (nothing to seek in unloaded content) or implicitly allowed (position is queued for resumption). Currently seek is allowed on Buffering because has_content() excludes only Idle.

- **Spec:** Home Assistant media_player: seek is a transport verb; its validity in Buffering is not explicit. RFC 3986 and streaming specs: position in unstarted streams is undefined.
- **Notes:** Current behavior: seek is allowed on Buffering. Spec is ambiguous. Either reject with NothingPlaying, or document that seek is queued and applied on play(). Test: LaunchApp → SeekRelative → Play, verify position is maintained.

### `source-selection-clears-app-position` — phase-2 — ✅ TDD-ready

Selecting a source (e.g., switching to HDMI 1) correctly clears the app and resets position. This is implemented, but no test validates the full sequence of launching an app on one source, selecting a different source, then returning to the first.

- **Spec:** Home Assistant: SelectSource implies switching input hardware; any app context becomes invalid. media_player entity: current_source is mutually exclusive with app-based playback.
- **Notes:** Behavior is correct (current_app set to None, playback set to Idle). Missing integration test: Launch Netflix on implicit source, SelectSource(HDMI1), verify current_app==None, PlaybackState==Idle, position==0. Then SelectSource back to app context (if supported).

### `volume-step-default-constant` — phase-2 — ✅ TDD-ready

volume_up() and volume_down() accept a step parameter, but there is no defined default step size (e.g., 5% increments) used by remote control or voice commands. Home Assistant defines step_percentage; cave-home-display does not.

- **Spec:** Home Assistant media_player: step_percentage attribute (typically 5 or 10). HDMI-CEC / IR specs: standard volume increment is ~10% (255 / 26 = 9.8).
- **Notes:** Currently each caller chooses the step. Recommend adding pub const DEFAULT_VOLUME_STEP: u8 = 5; or 10. Then add MediaCommand::VolumeUp / VolumeDown (no parameter) that uses the default. Test: repeated VolumeUp from 0 should reach 100 in ~10 commands.

### `stop-clears-position-only` — phase-2 — ✅ TDD-ready

Stop clears the position to 0, and moves to Stopped state. No test validates that the app/source are NOT cleared (unlike SelectSource or PowerOff). Spec clarity: Stop is a transport control, not a context switch.

- **Spec:** Home Assistant media_player: Stop stops playback at current position; does not change context. Differs from SelectSource (context switch) or PowerOff (full reset).
- **Notes:** Behavior is correct (only playback and position change). Missing test: Launch Netflix, Play, Stop, verify current_app still == 'netflix', position == 0, playback == Stopped, then Play resumes.

### `mute-does-not-affect-audible-on-power-off` — phase-2 — ✅ TDD-ready

When the TV powers off, mute state is preserved (not mentioned). When it powers back on, is sound muted or unmuted? Charter requires clarity: does power-off reset mute to unmuted?

- **Spec:** Home Assistant media_player: power state and mute are independent; spec does not define mute behavior on power transitions. ADR-028: household clarity required.
- **Notes:** Current code: PowerOff does not reset mute, so mute persists across power cycles. Test: Set volume 50, SetMute(true), PowerOff, PowerOn, verify volume is 50 but still muted. Confirm this is desired (or add PowerOff → SetMute(false)).

### `next-previous-on-stopped-state` — phase-2 — ✅ TDD-ready

Next and Previous are rejected on Stopped state (has_content() returns true, so they are allowed). But semantics are unclear: does Next on a stopped playlist restart it or advance to next without playing? Home Assistant is ambiguous.

- **Spec:** Home Assistant media_player: next/previous are transport verbs; stop disengages transport control. Stopped state: no current position, playlist is not 'active'.
- **Notes:** Current behavior: Next on Stopped → resets position to 0, transitions to Playing. Test should verify this is correct or document the deviation. Test: LaunchApp, Play, Stop (position → 0), Next, verify position stays 0 and playback is Playing (i.e., next item plays immediately).

### `pause-on-stopped-is-noop` — phase-2 — ✅ TDD-ready

Pause on Stopped state is a no-op (pause() returns Stopped unchanged). This is correct, but the test coverage does not explicitly verify the no-op behavior for Stopped specifically.

- **Spec:** Home Assistant media_player: Pause is only valid on Playing or Buffering; other states are no-ops (idempotent).
- **Notes:** Behavior is correct but implicit. Add explicit test: Display.apply(Pause) when playback == Stopped, verify playback remains Stopped and no error.

### `status-sentence-paused-app-context` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

status_sentence() returns 'The TV is paused' generically, with no indication of which app is paused (e.g., 'Netflix paused'). Charter §6.3 requires household clarity; current behavior loses app context in the status.

- **Spec:** Charter §6.3 / ADR-028: Status must be household-legible; generic 'paused' does not tell the household WHICH app is paused (critical for multi-app households).
- **Notes:** Already implemented as generic status, but specification gap: should status sentence include app name (e.g., 'Netflix is paused')? Requires Display to return dynamic &str (not &'static). Defer to phase-2 for richer status model. Currently test expects generic label.
