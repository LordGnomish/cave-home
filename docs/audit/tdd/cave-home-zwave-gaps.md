# TDD gap analysis — `cave-home-zwave`

- **Upstream:** Silicon Labs Z-Wave Command Class Specification (public, SDS13781 family)
- **License class:** permissive
- **Port method:** spec-based clean-room implementation from public Z-Wave CC specification; node-zwave-js MIT was ecosystem reference per ADR-001 but no GPL/AGPL source read
- **Existing cave tests (agent count):** 58
- **Missing behaviors identified:** 11 (11 strict-TDD-ready)

> Generated by the spec-based gap-analysis pass (read-only). For
> `gpl-agpl-cleanroom` upstreams no upstream source/tests were consulted —

**Summary.** cave-home-zwave is a spec-based, clean-room Z-Wave Command Class engine (58 tests). Phase 1 MVP implements 11 CCs (Basic, Binary/Multilevel Switch, Binary/Multilevel Sensor, Meter, Color, Thermostat Mode/Setpoint, Configuration, Notification, Battery) plus encoding/decoding, fixed-point value codec, addressing, and EN/DE/TR UX rendering. Transport (Serial API, security, OTA) and controller management are Phase 1b-deferred per Charter 3.2. Missing behaviors cluster in two areas: (1) Phase-1b deferred but spec-complete CCs (Multi Channel, Wake Up, Association, Door Lock) that require transport layer support, and (2) Phase-2 enhancements to existing CCs that add validation, duration fields, scale mapping, or type-specific constraints per Z-Wave spec but are not critical for MVP.

## Missing behaviors

### `multi-channel-cc` — phase-1b — ✅ TDD-ready

Multi Channel Command Class (0x60) encapsulation and endpoint routing: ability to decode/encode reports that route commands to specific endpoints via the Multi Channel Encapsulation command, allowing a single physical device with multiple logical endpoints to be individually addressed

- **Spec:** Z-Wave CC Specification SDS13781 family: Multi Channel Command Class (0x60), Set/Get/Report; multi-endpoint device routing
- **Notes:** Phase 1 MVP explicitly defers this as hardware/network-bound device enumeration (parity.manifest [[unmapped]]). Spec-defined public behavior with no upstream source dependency makes it TDD-ready. Endpoint 0 addressing exists in address module but CC-level encapsulation not yet modeled.

### `wake-up-cc` — phase-1b — ✅ TDD-ready

Wake Up Command Class (0x84) interval reporting and device wake scheduling: ability to decode/encode Wake Up Interval Set/Report and Wake Up No More Information commands for battery-powered devices to announce their wake interval and receive queued commands

- **Spec:** Z-Wave CC Specification SDS13781 family: Wake Up Command Class (0x84), Interval Set/Report, No More Information
- **Notes:** Deferred as hardware/transport-bound per Charter 3.2 parity.manifest. Battery-aware devices use this CC; test can be written from spec byte sequences. Currently only Battery CC level (0x80) is modeled, not wake interval interaction.

### `association-cc` — phase-1b — ✅ TDD-ready

Association Command Class (0x85) group membership and device linking: ability to decode/encode Set/Get/Remove commands to configure which devices a node sends unsolicited reports to, supporting multi-endpoint association via Multi Channel Association (0x8E)

- **Spec:** Z-Wave CC Specification SDS13781 family: Association CC (0x85) Set/Get/Remove/Supported Get; Multi Channel Association (0x8E)
- **Notes:** Deferred as controller/network-management-bound (parity.manifest [[unmapped]]). Pure CC protocol behavior with no upstream dependency. Would allow cave-home to configure device-to-device automation once transport layer lands.

### `door-lock-cc` — phase-2 — ✅ TDD-ready

Door Lock Command Class (0x62) lock state and secure access: ability to decode/encode lock/unlock Set/Report commands with battery status and operation type, handling both secured and unsecured lock modes

- **Spec:** Z-Wave CC Specification SDS13781 family: Door Lock CC (0x62), Set/Get/Report; lock state (locked/unlocked), battery status
- **Notes:** Not yet in Phase 1 MVP scope. Smart lock is a valid Phase 2 device type (household security pillar per Charter). Spec byte sequences are public; test can assert correct lock-state projection onto Value model once variant is added.

### `switch-binary-duration` — phase-2 — ✅ TDD-ready

Binary Switch CC duration/fade: Set command with optional duration byte (0x00-0xFF or 0xFF for factory defaults) to control on/off transition time, allowing gradual relay fade effects

- **Spec:** Z-Wave CC Specification SDS13781 family: Binary Switch CC (0x25) Set v2+ with optional duration field
- **Notes:** Currently Command::BinarySwitchSet(bool) takes only state without duration. Spec version 2+ adds optional byte 2 for duration. Could extend to Optional<u8>. Test verifies default 0xFF factory mapping and fallback for duration-less devices.

### `switch-multilevel-duration` — phase-2 — ✅ TDD-ready

Multilevel Switch CC duration/fade: Set command with optional duration field to support gradual dimming over specified seconds, and Start/Stop Level Change with duration for smooth motor/blind operation

- **Spec:** Z-Wave CC Specification SDS13781 family: Multilevel Switch CC (0x26) Set v4+ with duration; Start/Stop Level Change with optional duration
- **Notes:** MultilevelSwitchSet currently takes only level; duration not yet modeled. Start/Stop Level Change exists but optional duration field not decoded. Spec encodings are public; roundtrip tests can verify fade behavior.

### `sensor-multilevel-scale-validation` — phase-2 — ✅ TDD-ready

Multilevel Sensor CC scale validation per sensor type: reject or map out-of-spec scale values (e.g. scale 2 or 3 for air temperature which only defines scale 0/Celsius and 1/Fahrenheit) to prevent invalid unit interpretation

- **Spec:** Z-Wave CC Specification SDS13781 family: Multilevel Sensor CC (0x31); sensor-type-specific scale constraints
- **Notes:** Currently sensor_decode accepts any scale 0-3 without validating against sensor_type. Temperature hardcodes 0->C, 1->F but never rejects 2 or 3. Strict validation would prevent silent misinterpretation of malformed reports.

### `meter-scale-quantity-mapping` — phase-2 — ✅ TDD-ready

Meter CC scale-to-quantity mapping: validate and map meter-type-specific scale values (e.g. electric 0x01 with scale 0=kWh, 1=kVAh, 2=W, 3=A) so energy vs power readings project onto correct Quantity::Energy or Quantity::Power without ambiguity

- **Spec:** Z-Wave CC Specification SDS13781 family: Meter CC (0x32); meter-type-specific scale encodings
- **Notes:** Code at line 509 hardcodes 'scale==0 ? Energy : Power' without validating meter type. Electric meter (0x01) has 4 scales; gas (0x02) has different scales. Missing validation allows invalid scale+type pairs to silently misproject onto wrong Quantity.

### `color-component-validation` — phase-2 — ✅ TDD-ready

Color Switch CC component id validation: define and validate component type ranges (0=warm white, 1=cold white, 2=red, 3=green, 4=blue, etc.) rather than accepting any u8 component id

- **Spec:** Z-Wave CC Specification SDS13781 family: Color Switch CC (0x33) component id field; standard 0-6 + manufacturer-specific 0x07
- **Notes:** Code accepts any u8 component id without bounds checking. Spec defines 0-6 as named (warm/cold white, RGB) + 0x07 for OEM. Out-of-range values should be rejected per crate error convention.

### `notification-type-event-validation` — phase-2 — ✅ TDD-ready

Notification CC type+event validation: validate event id against notification type (e.g. type 0x01 Smoke Alarm allows events 0x00-0x03, not arbitrary values) and map known events to semantic labels

- **Spec:** Z-Wave CC Specification SDS13781 family: Notification CC (0x71); type 0x01 Smoke events 0x00-0x03, type 0x02 CO has different events
- **Notes:** Currently accepts any u8 event value without validation against notification_type. Type 0x01 Smoke has 4 valid events; type 0x02 CO has different events. Known types/events could be modeled like ThermostatMode.

### `setpoint-type-validation` — phase-2 — ✅ TDD-ready

Thermostat Setpoint CC type validation: validate setpoint type against known types (1=heating, 2=cooling, 3=furnace, etc.) and ensure temperature scale consistency across Get/Report pairs for same setpoint type

- **Spec:** Z-Wave CC Specification SDS13781 family: Thermostat Setpoint CC (0x43); setpoint-type field defines 0x00 (not supported) through 0x08+ (manufacturer-specific)
- **Notes:** Code accepts any u8 setpoint_type without validation. Spec defines types 0x00-0x08+. Out-of-range types should be rejected like ThermostatMode does.
