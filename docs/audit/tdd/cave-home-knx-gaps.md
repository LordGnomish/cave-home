# TDD gap analysis — `cave-home-knx`

- **Upstream:** Public KNX standard (DPT/APCI); xknx (MIT, public-behavior reference); KNXd (GPL-3.0, clean-room recusal)
- **License class:** gpl-agpl-cleanroom
- **Port method:** spec-based (public KNX DPT/APCI tables) + xknx-MIT public-behavior reference
- **Existing cave tests (agent count):** 62
- **Missing behaviors identified:** 12 (12 strict-TDD-ready)

> Generated by the spec-based gap-analysis pass (read-only). For
> `gpl-agpl-cleanroom` upstreams no upstream source/tests were consulted —
> behaviors are derived from public protocol/spec knowledge only.

**Summary.** cave-home-knx is a fully-spec'd Phase-1 KNX codec engine (62 tests, all passing). All implemented behaviors (DPT 1/2/3/5/6/7/8/9/12/13/14/16, addresses, APDU framing, UI labels) are covered by round-trip and error-case tests. However, the test suite lacks focused coverage of: (1) spec-mandated precision contracts (DPT 9 lossiness bands per exponent, DPT 5 rounding tie-break semantics); (2) frame/service invariants (Read never has payload, small-payload optimization boundary rules); (3) DPT 4.x (listed in spec as deferred); (4) exhaustive length-rejection for all DPTs; (5) encoding notation preservation (address style round-trip); (6) integration scenarios (multi-telegram flows, edge cases in time/date year mapping). The crate is clean-room (KNXd NOT read), dependency-free, and well-architected; these gaps are test-design (not implementation) gaps — opportunities for stricter TDD.

## Missing behaviors

### `dpt4-4bit-signed` — phase-1b — ✅ TDD-ready

DPT 4.x — 4-bit signed integer (two's complement, -8..=7). Encode/decode with range validation and wrong-length rejection.

- **Spec:** KNX Standard DPT 4.x main type (4-bit signed field, used for relative HVAC adjustments; core residential datapoint)
- **Notes:** Listed in parity.manifest as unmapped/deferred; pub spec says 4.x exists but not yet in crate. Straightforward 4-bit signed codec, similar to dpt3 but with signed interpretation instead of direction. A true spec gap: the spec lists it, the codec does not implement it.

### `address-range-validation` — phase-1b — ✅ TDD-ready

Comprehensive integration test for address field edge cases: group addresses at boundary values (0/0/0 vs 31/7/255 in 3-level; 31/2047 in 2-level; bare 65535), and individual addresses with area=0 or line=0 (valid per spec, often treated as broadcast).

- **Spec:** KNX Spec §8.3.3 Group Address Structure: main 5-bit (0..31), middle 3-bit (0..7), sub 8-bit (0..255); §8.3.1 Individual Address: area 4-bit (0..15), line 4-bit (0..15), device 8-bit (0..255). Valid ranges allow zero in all fields.
- **Notes:** Current tests parse/roundtrip valid addresses but do not exhaustively verify boundary semantics (e.g., area=0 broadcast interpretation, reserved group addresses 0/0/x for management). Strict TDD: a test that verifies every field at min/max + zero does not panic and encodes consistently.

### `dpt9-lossless-region` — phase-1b — ✅ TDD-ready

DPT 9 encoding precision: verify that certain ranges are lossless (e.g., 0.01 multiples in a defined band) and identify the quantization quantum in the lossy region (±671088.64). Encode-decode tolerance should be specified and validated for every representable quantum.

- **Spec:** KNX DPT 9.x spec: value = 0.01 * M * 2^E. Exponent 0..15 determines step size. In exp=0, step=0.01. In exp=4, step=0.16. Lossless region: approximately ±40.96 (where exp <= 1). Lossy beyond that.
- **Notes:** Current tests check a range sweep with hand-tuned tolerance but do NOT have a formal spec of the lossless band or explicit quantum-per-exponent tests. A strict TDD test: for each exponent 0..15, verify the step size = 0.01 * 2^exp, and round-trip a value at step boundaries with zero tolerance in lossless range.

### `dpt5-rounding-tie-break` — phase-1b — ✅ TDD-ready

DPT 5 scaling / angle encode: round-half-up is specified (KNX standard); verify that 50.0% encodes as 128, 50.2% as 129, 49.8% as 127 (confirming half-way cases round away from zero, not banker's rounding).

- **Spec:** KNX DPT 5.x spec: value = round(raw / 255 * range). Rounding method: round-half-away-from-zero (KNX default, matching ETS). Current code uses Rust's f64::round() which is also half-away-zero; test should verify the exact tie-break semantics for 50.0%, 1.96%, 98.04%.
- **Notes:** The crate's encode_scaling test checks that 50.0 % -> 128 via 127.5 round-up, but does not exhaustively test the tie-break rule for other fractional boundaries. A strict test: 50.0 % (127.5 rounds up), 1.96078% (5 rounded from 5.000...), 98.03922% (250 rounded from 250.000...).

### `group-service-read-response-pairing` — phase-1b — ✅ TDD-ready

Integration: a GroupValueRead telegram on a group should semantically pair with a GroupValueResponse on the same source/destination pair (i.e., protocol pattern validation). Test that a Read APDU (0b0000) is always length-2 (no payload), and a Response/Write (0b0001/0b0010) can carry a payload.

- **Spec:** KNX APCI group services §7.3: Read (0b0000), Response (0b0001), Write (0b0010). Read always has no payload. Response and Write may carry data. This is a service invariant, not a datapoint invariant.
- **Notes:** Current tests verify encode/decode round-trips but do not have a dedicated test checking that GroupService::Read ALWAYS produces a 2-byte APDU and cannot carry a payload. This is a spec invariant: Read never has data (only the 2-byte control frame).

### `small-payload-optimization-coverage` — phase-1b — ✅ TDD-ready

Exhaustive test of the ≤6-bit small-payload optimization: verify that dpt1 (1 bit), dpt2 (2 bits), dpt3 (4 bits) values are packed into the APDU control byte (bits 5..0 of byte1) and NOT appended; test that a dpt3::encode value uses the optimized form, not a 1-byte appended payload; test that decoding a 2-byte APDU with non-zero bits 5..0 correctly recovers the small value.

- **Spec:** KNX APDU framing §7.2: small payloads (≤6 bits) occupy bits 5..0 of byte1 when APDU is exactly 2 bytes. Longer payloads clear those bits and append bytes. Test boundary: dpt3 is 4 bits (qualifies), hypothetical dpt8 (16 bits) does not.
- **Notes:** Current tests verify round-trips of small and large payloads but do NOT have a focused test verifying the optimization rule for EVERY small DPT type. Test: dpt1::encode(true) -> 0x01 -> write_small(src, dst, 0x01) -> APDU = [0x00, 0x80 | 0x01]; NO third byte.

### `dpt-payload-length-contract` — phase-1b — ✅ TDD-ready

Each DPT's decode function MUST reject payloads of incorrect length. Test suite should verify that every DPT (1..16, including 10/11) rejects wrong-length inputs: dpt1/2/3 reject anything >1 byte, dpt5/6 reject !=1, dpt7/8 reject !=2, dpt9/10/11 reject !=2/3/3, dpt12/13 reject !=4, dpt14 reject !=4, dpt16 reject !=14.

- **Spec:** KNX DPT spec: each datapoint has a fixed wire width. Codec contract: decode(bytes) must validate len(bytes) == expected_width and return Err if not. No panics on malformed input.
- **Notes:** Current tests check some wrong-length cases but not exhaustively for all DPTs. A strict TDD test should be a table-driven test: for each (dpt, correct_len), test decode with len-1, len, len+1 and verify only len succeeds. Already partially covered but lacks systematic coverage.

### `action-enum-exhaustiveness` — phase-1b — ✅ TDD-ready

The Action enum (Light, Blinds, Temperature) with its describe(Lang) method should be tested to ensure every (action, language) pair is in ({{Light on/off, Blinds up/down, Temperature N°}} × {{En, De, Tr}}) = 9 combinations, and that the result is never empty and never contains protocol jargon (DPT, APCI, group address, etc.).

- **Spec:** ADR-007 / Charter §6.3: grandma-friendly EN/DE/TR descriptions. Every household action must have a translation in all three languages. No technical terms surface.
- **Notes:** Current tests verify localization for light/blinds/temperature and check that no jargon appears. A strict TDD test: enumerate all 9 combinations explicitly, verify each non-empty, and spot-check the output for common jargon keywords (DPT, 0x, byte, KNX).

### `telegram-frame-invariants` — phase-1b — ✅ TDD-ready

GroupTelegram struct carries four fields (source, destination, service, payload); test that encode_apdu/decode_apdu preserve all four consistently. Verify that a telegram with an explicit source/destination can be encoded then decoded and recovers the same values (round-trip source/destination).

- **Spec:** KNX spec: a telegram is identified by (source_individual_address, destination_group_address, service_code, data). The APDU carries only the service and data; source/destination are carried in separate fields per KNXnet/IP frame. This codec models them in the struct; round-trip must recover all four.
- **Notes:** Current tests verify APDU encode/decode but do not explicitly test that source and destination survive a round-trip through the codec. A strict test: write a telegram with src=1.1.5, dst=1/2/3, encode_apdu, then decode_apdu with the SAME src/dst, and assert both survive unchanged.

### `dpt10-day-field-semantics` — phase-1b — ✅ TDD-ready

DPT 10 Time day field (bits 7..5) encodes day-of-week 0..7: 0=unspecified, 1=Monday, ..., 7=Sunday. Test that day field is correctly extracted and the reserved bit patterns (if any) are handled. Verify that day in encode must be 0..7 (NOT 1..7, zero is valid).

- **Spec:** KNX DPT 10.001 spec: day field 3 bits (0..7); 0=no day, 1=Mon, 2=Tue, ..., 7=Sun. Encode/decode must accept day=0 (meaning 'no specific day').
- **Notes:** Current integration tests (dpt_time_date.rs) verify this with an explicit test (dpt10_time_midnight_no_day_is_all_zero). A stricter TDD test should verify day=0..7 all encode without error and day=8 is rejected; also verify each day 1..7 decodes back correctly.

### `dpt16-utf8-latin1-boundary` — permanent — ✅ TDD-ready

DPT 16 string codec: test the edge between valid UTF-8 and invalid-but-latin1 byte sequences. Verify that the codec REJECTS latin-1 bytes (0x80..0xFF) that are not valid UTF-8, and only accepts valid UTF-8 (including ASCII, which is a UTF-8 subset).

- **Spec:** KNX DPT 16.x spec allows latin-1; cave-home codec doc-comment (src/dpt.rs:684) explicitly rejects non-UTF-8: 'we accept ASCII / UTF-8; latin-1 high bytes that are not valid UTF-8 are rejected rather than silently mangled'.
- **Notes:** Current test encodes/decodes ASCII and UTF-8. Test is missing: attempt to decode a 14-byte payload with byte 0xFF (valid latin-1, invalid UTF-8) and verify it returns Err, not a mangled string. This enforces the documented behavior.

### `address-style-preservation` — permanent — ✅ TDD-ready

GroupAddress::parse preserves the input notation style (Free / TwoLevel / ThreeLevel) and Display round-trips it. Test that parse('1/2/3') -> display() = '1/2/3', parse('1/500') -> display() = '1/500', parse('12345') -> display() = '12345' (not converted to a different notation).

- **Spec:** KNX spec: group address notation is installer-chosen. cave-home docs (address.rs:24): 'The notation only changes how the number is split for humans.' Round-trip is in user-facing code (config, UI) and must preserve user intent.
- **Notes:** Current tests parse and display individual addresses and group addresses. A focused test is missing: parse('5/500') in 2-level, display it, and verify output is '5/500' not '5/0/500'. Similarly for all three styles. Prevents silent notation conversion.
