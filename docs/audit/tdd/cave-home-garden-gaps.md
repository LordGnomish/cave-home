**Summary.** cave-home-garden is the plant-care decision engine (Phase 1 MVP) shipping the frost/light/soil/temperature/season classifiers, combined care-advice engine, and EN/DE/TR UX. All core logic is fully implemented and tested (36 tests). Phase-1b and Phase-2 gaps are I/O and integration boundaries: outdoor sensor adapters (Zigbee/BLE/ESPHome), live weather feeds, plant-database import, cave-home-water integration, cave-home-core entity surface, robot-mower control, and multi-day forecast extrapolation are documented ADR-029 deferred items.

## Missing behaviors

### `soil-moisture-sensor-adapter-zigbee` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Outdoor soil-moisture sensor adapters over Zigbee, BLE, or ESPHome bridges mapping raw probe readings (0-100% ADC or m3/m3) onto Conditions.soil_moisture_percent

- **Spec:** parity.manifest [[unmapped]] 'Outdoor soil-moisture / light / temperature sensor adapters'; ADR-029 Phase 1b
- **Notes:** Hardware-bound. The engine accepts soil_moisture_percent as a pure input; adapters translate vendor probe protocols (Zigbee moisture sensors, BLE GATT, ESPHome REST) into this percentage. No decision logic change to engine — purely I/O mapping.

### `light-sensor-adapter-lux-calibration` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Light sensor adapters (Zigbee, BLE, ESPHome) mapping raw lux readings or coarse bands onto LightInput (measured lux or discrete LightBand::Dark/Dim/Bright/FullSun)

- **Spec:** parity.manifest [[unmapped]] 'Outdoor soil-moisture / light / temperature sensor adapters'; ADR-029 Phase 1b
- **Notes:** Hardware-bound. The engine accepts LightInput as a pure input. Adapters translate vendor sensor reports (lux meters, brightness classification) into Lux(f64) or Band enum. Calibration constants per sensor model may vary.

### `weather-forecast-feed-integration` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Live weather-forecast feed (Davis Vantage, Ecowitt, Netatmo Weather) supplying daily/hourly forecast data; extraction of overnight minimum temperature (forecast_min_c) for frost classification

- **Spec:** parity.manifest [[unmapped]] 'Live weather-forecast feed'; ADR-029 Phase 1b; Home Assistant weather integrations Apache-2.0 (husqvarna_automower, worx_landroid, weather)
- **Notes:** Network-bound. Engine is given forecast_min_c as a scalar input. A Phase-1b adapter fetches live forecasts (via HTTP/REST to weather APIs or local ESPHome stations) and feeds the overnight low into FrostRisk::classify. No change to engine itself.

- **Notes:** Data/IO concern. The Plant and CareProfile models already accept arbitrary profiles via Plant::custom(name, profile). Import logic (parsing CSV/JSON, validating thresholds, storage) is deferred; the engine itself needs no change.

### `cave-home-water-recommendation-routing` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

### `multi-day-forecast-frost-dating` — phase-2 — ✅ TDD-ready

Sub-daily and multi-day weather-forecast extrapolation (e.g. 'first hard frost date in the next 7 days'), smoothing a sequence of overnight minima into long-horizon guidance for hardening-off or season-end harvest planning

- **Spec:** parity.manifest [[unmapped]] 'Sub-daily / multi-day weather extrapolation for frost timing'; ADR-029 Phase 2
- **Notes:** Phase 2 refinement. Engine currently handles a single forecast_min_c per call (correct for 'frost tonight?'). A future module could accept a multi-day forecast sequence [min_c, ...] and apply the same thresholds over a rolling window to produce 'first frost date' or 'last frost date' summaries. TDD-ready: spec is threshold-based, same plant model.

### `humidity-and-vapor-deficit-assessment` — phase-2 — ✅ TDD-ready

Assess ambient humidity and vapor-pressure-deficit (VPD) vs. plant's ideal RH band; report 'too dry air' or 'too humid' (relevant to tropical plants, succulents, and powdery-mildew risk assessment)

- **Spec:** Not currently in spec or parity.manifest. Extension beyond Phase 1 MVP (frost/light/soil/temperature). Public horticulture references: VPD thresholds for common crops.
- **Notes:** Phase 2 extension. The crate is designed as a pure function; adding RH % input to Conditions and a humidity assessment enum + message similar to TempFit would be straightforward. Specs exist (e.g. 'optimal RH for tomatoes 60–80%'). No upstream HA integration blocking it (this is novel logic).

### `southern-hemisphere-plant-zone-mapping` — phase-2 — ⏸ not-yet (needs impl-deps / test-after)

Map common plant hardiness zones (e.g. USDA zones, UK RHS zones) to frost-sensitivity classes (Tender/HalfHardy/Hardy) and auto-populate CareProfile.frost_sensitivity based on plant species and geographic zone

- **Spec:** Light references: RHS / extension-service guidance; parity.manifest spec_sources list hardiness zones as implicit input to frost_sensitivity
- **Notes:** Data/reference concern. RHS or USDA zone listings are external reference data. The engine accepts FrostSensitivity as enum input; a Phase-2 module could offer a zone-to-sensitivity mapper for household import flows. No engine change needed.
