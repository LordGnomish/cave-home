**Summary.** cave-home-zigbee is a clean-room Zigbee 3.0 coordinator (121 tests, 3.6 kLOC Phase 1 MVP). Upstream is GPL-3.0 (zigbee2mqtt) but is not read — implementation is from public Zigbee 3.0 / ZCL specs, EZSP UG100, deCONZ protocol, and zigbee-herdsman public API docs. Phase 1 covers transport (EZSP/deCONZ), ZCL Foundation (Read/Write/Report), Groups/Scenes clusters, pairing (traditional join + InstallCode), OTA queue, and routing table. Missing behaviors are primarily Phase 1b / Phase 2 features documented in parity.manifest.toml: ZCL Foundation commands (Default Response, Discover Attributes, Read Reporting Config), OTA image block transfer codec, scene extension-field parsing, Groups response payloads, and security features (key rotation, network steering timeout, InterPAN Touchlink layer, device re-announcement). Most are TDD-ready from spec; 3 require integration testing or MAC-layer work. Strict clean-room port — no upstream source consulted.

## Missing behaviors

### `zcl-default-response` — phase-1b — ✅ TDD-ready

ZCL Default Response (0x0b) command encode/decode — profile-wide command allowing servers to report unsupported cluster-specific commands with status codes (ZCL §2.4.6 Table 2-9)

- **Spec:** ZCL §2.4.6 Default Response payload; Zigbee 3.0 §2.4 Table 2-3 command 0x0b
- **Notes:** Phase 1 implements Read/Write/Report but omits Default Response. This is a Foundation command that servers should send when receiving unknown cluster-specific commands. TDD test can be written now from ZCL spec without upstream source: (command_id: u8, status: u8) encode/decode round-trip.

### `zcl-discover-attributes` — phase-1b — ✅ TDD-ready

ZCL Discover Attributes (0x0c) and Response (0x0d) — allows clients to enumerate cluster attributes dynamically (ZCL §2.4.12)

- **Spec:** ZCL §2.4.12 Discover Attributes payload and Discover Attributes Response; Table 2-3 commands 0x0c and 0x0d
- **Notes:** Phase 1 MVP does not implement attribute discovery. This is a Foundation command pair for introspecting cluster schema. TDD test can be written from ZCL §2.4.12.1 format: encode (start_attr_id: u16, max_attrs: u8), decode (discovery_complete: bool, attr_id: u16, data_type: ZclDataType) list.

### `zcl-configure-reporting-response` — phase-1b — ✅ TDD-ready

ZCL Configure Reporting Response (0x07) — server acknowledges Configure Reporting with per-attribute status records (ZCL §2.4.7.3)

- **Spec:** ZCL §2.4.7.3 Configure Reporting Response payload; Table 2-3 command 0x07
- **Notes:** Phase 1 sends Configure Reporting (0x06) but does not model/test the response (0x07) format: (status: u8, direction: u8, attr_id: u16) records. TDD can verify round-trip decode from spec.

### `zcl-read-reporting-config` — phase-1b — ✅ TDD-ready

ZCL Read Reporting Configuration (0x08) and Response (0x09) — allows clients to query the server's current reporting configuration per attribute (ZCL §2.4.8 / §2.4.9)

- **Spec:** ZCL §2.4.8 Read Reporting Configuration payload and §2.4.9 Response; Table 2-3 commands 0x08 and 0x09
- **Notes:** Phase 1 MVP does not read back reporting configuration. This is a Foundation command pair for introspecting active subscriptions. TDD test: encode request with (direction: u8, attr_id: u16) list, decode response with (status: u8, direction, attr_id, data_type, min_interval, max_interval, report_value_delta) records.

### `ota-query-next-image-request` — phase-1b — ✅ TDD-ready

OTA Query Next Image Request (0x01) payload decode — extract manufacturer_code, image_type, file_version from device's request to populate OtaJob (ZCL §11.4.1)

- **Spec:** ZCL §11.4.1 Query Next Image Request payload format (u16 mfr, u16 type, u32 version)
- **Notes:** Phase 1 OtaQueue accepts enqueued OtaJob but does not decode inbound Query Next Image Request frames. Clean-room TDD: encode/decode (manufacturer_code: u16, image_type: u16, current_file_version: u32) with optional hardware_version field.

### `groups-command-response-payload` — phase-1b — ✅ TDD-ready

ZCL Groups cluster command response payloads (0x00-0x05) — Add Group Response, View Group Response, Get Group Membership Response, Remove Group Response status encoding (ZCL §3.6.2.2)

- **Spec:** ZCL §3.6.2.2.1–2.2.5 Groups command responses — (status: u8, group_id: u16) and (status, groups: u16[]) tuples
- **Notes:** Phase 1 GroupsCluster has add/view/remove/membership logic but does NOT encode/decode the cluster-specific command responses (0x00–0x05 response payloads). TDD: (status: u8, [group_id: u16] or [groups: u16[]]) encode/decode per command.

### `ota-image-block-transfer` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

OTA Image Block Request (0x05) / Response — handle firmware download block-by-block with sequence, size, and CRC tracking; Upgrade End Request (0x06) / Response status codes (ZCL §11.4)

- **Spec:** ZCL §11.4.5 Image Block Request/Response payload format; ZCL §11.4.6 Upgrade End Request/Response; Table 11-2 command IDs 0x05, 0x06
- **Notes:** Already partially implemented: Phase 1 ships command_id constants and OtaQueue, but Phase 1b payload encode/decode and signature verification are explicitly deferred. Implementation would require ECDSA validator + block reassembly; spec-only TDD is feasible but integration tests require live OTA provider.

### `network-steering-permit-join-timeout` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

Network steering permit-join window timeout behavior: automatically close the network after duration expires (Zigbee 3.0 §2.5.5.5.4; coordinator should issue LEAVE after timer expires)

- **Spec:** Zigbee 3.0 §2.5.5.5.4 Network Steering — NLME-PERMIT-JOINING.request with timeout behavior
- **Notes:** NetworkSteering tracks open_window and duration but does NOT auto-close the network when the timer fires. Testing requires async tokio::time coordination. Phase 1b feature once coordinator's main event loop is stable.

### `interpan-touchlink-frame-layer` — phase-1b — ⏸ not-yet (needs impl-deps / test-after)

InterPAN / Touchlink commissioning frame layer — Zigbee Profile frame (MAC PAN ID broadcast, short MAC source, no 802.15.4 ack) for over-the-air Touchlink without join-network prerequisite (BDB §10.1.1)

- **Spec:** Zigbee Base Device Behavior §10.1 Touchlink; IEEE 802.15.4-2020 §4.1.2 short addressing for InterPAN frames
- **Notes:** Phase 1 ships TouchlinkMode enum to signal coordinator intent but does not implement the InterPAN frame layer itself. This is a separate MAC-layer primitive required for Touchlink target role. Deferred to Phase 1b.

### `scene-extension-field-encoding` — phase-2 — ✅ TDD-ready

ZCL Scenes cluster extension-field-set encoding/decoding — per-cluster attribute snapshots in scene storage (ZCL §3.7.2.5 Extension Field Set format). Currently accepted as raw Vec<u8>.

- **Spec:** ZCL §3.7.2.5 Extension Field Set — nested (cluster_id: u16, length: u8, attributes: [...]) tuples storing attribute values per cluster
- **Notes:** Phase 1 stores scene extension fields as opaque Vec<u8>; no codec. TDD can verify round-trip of (cluster_id: u16, attribute payload) tuples. Deferred until Portal scene UI matures.
